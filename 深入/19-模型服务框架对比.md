# 深入 19 · 模型服务框架对比

> [← 返回目录](../README.md)  ·  相关：[深入 01 · TTFT 与吞吐](01-首包延迟与吞吐的影响因素.md)  ·  [深入 05 · 容量规划](05-LLM推理服务的容量规划.md)  ·  [第 5 章 · 推理服务的可靠性工程](../知识/05-AI推理服务的可靠性工程.md)

> [!NOTE]
> vLLM、TGI、SGLang、TensorRT-LLM 在全书各章零星出现。这一章集中对比，回答"到底该选哪个"——从 SRE 关心的维度出发：TTFT、吞吐、显存效率、cache 支持、容错、运维成本。

---

## 0. 一句话定位

| 框架 | 一句话 | 典型用户 |
|---|---|---|
| **vLLM** | 开源社区的事实标准，PagedAttention 发明者 | 90% 的自建推理场景 |
| **SGLang** | 吞吐之王，RadixAttention + 结构化生成 | 高吞吐场景、Agent 推理 |
| **TGI (Text Generation Inference)** | HuggingFace 官方，生态最全但节奏慢 | HF 生态深度用户 |
| **TensorRT-LLM** | NVIDIA 官方，极致优化但门槛高 | 追求极致性能 + NVIDIA 全栈用户 |

## 1. 核心维度对比

### 1.1 性能

| 维度 | vLLM | SGLang | TGI | TensorRT-LLM |
|---|---|---|---|---|
| **TTFT（首 token 延迟）** | 好 | **优秀** | 好 | **最优** |
| **吞吐（tokens/s）** | 好 | **最优**（RadixAttention 前缀共享） | 中 | **最优**（但需大量调优） |
| **显存效率** | 好（PagedAttention） | 好（RadixAttention，类似机制） | 好 | **最优**（图优化 + 算子融合） |
| **长 context (128k+)** | 支持 | **最优**（前缀缓存命中率高） | 支持 | 支持（需足够显存） |
| **Prefill 优化** | Chunked prefill | **RadixAttention** | 标准 | **In-flight batching** |

### 1.2 功能

| 维度 | vLLM | SGLang | TGI | TensorRT-LLM |
|---|---|---|---|---|
| **Continuous batching** | ✅ | ✅ | ✅ | ✅ |
| **Prefix caching** | ✅ (Automatic Prefix Caching) | ✅ (RadixAttention，自动 + 高效) | ❌ (仅 prompt caching via API) | ✅ (需手动管理) |
| **Structured output** | ✅ (guided decoding) | ✅ (原生 JSON/Regex 约束) | ✅ (grammar-based) | ✅ |
| **LoRA adapter 热加载** | ✅ | 实验性 | ✅ | ❌ |
| **Speculative decoding** | ✅ | ✅ | ❌ | ✅ (Draft model) |
| **Multi-node 推理** | ✅ (Ray/Pipeline) | ✅ | ❌ (需手动) | ✅ (MPI) |
| **量化支持** | AWQ/GPTQ/FP8 | AWQ/GPTQ/FP8 | AWQ/GPTQ/FP8 | **FP8/INT4 原生最优** |
| **流式输出** | ✅ | ✅ | ✅ | ✅ |
| **OpenAI compatible API** | ✅ (原生) | ✅ (原生) | ✅ (通过 HF TGI Messages API) | ✅ (通过 Triton) |

### 1.3 运维

| 维度 | vLLM | SGLang | TGI | TensorRT-LLM |
|---|---|---|---|---|
| **部署复杂度** | **低**（pip install + 一行启动） | **低** | 中（需配置较多） | **高**（需模型编译、引擎构建） |
| **模型支持广度** | **极广**（200+ 架构） | 广（常用架构全覆盖） | 广（HF hub 模型开箱即用） | 窄（需 ONNX/TRT 支持） |
| **新模型上线速度** | 最快（社区驱动） | 快 | 中（HF 官方节奏） | 最慢（需编译） |
| **K8s 部署** | ✅ (官方 Helm) | ✅ | ✅ | ✅ (Triton Inference Server) |
| **可观测性** | Prometheus metrics + 日志 | Prometheus metrics | Prometheus metrics | **Triton 级**（最丰富） |
| **社区活跃度** | **最高** | 快速上升 | 高 | 中（NVIDIA 驱动） |
| **文档质量** | 好 | 改善中 | 好 | 好（但假设读者懂 CUDA） |

### 1.4 容错与稳定性

| 维度 | vLLM | SGLang | TGI | TensorRT-LLM |
|---|---|---|---|---|
| **OOM 处理** | 自动降级（减少 max_num_seqs） | 自动降级 | 重启 | 编译时确定，运行时 OOM = crash |
| **KV cache 抢占** | ✅ 支持（自动驱逐 + 重算） | ✅（RadixAttention 的 LRU 驱逐） | 部分支持 | 受限于显存规划 |
| **请求超时处理** | ✅ (可配) | ✅ | ✅ | ✅ (Triton 层) |
| **优雅关闭** | ✅ | ✅ | ✅ | ✅ |
| **故障恢复** | 外部（K8s 重启） | 外部 | 外部 | 外部 |

## 2. SRE 选型决策树

```
要自建推理服务
  │
  ├─ 用 NVIDIA 全栈 + 追求极致性能？
  │   └─ 是 → TensorRT-LLM（前提：团队有 CUDA 能力，能接受 1-2 周模型上线周期）
  │
  ├─ 主要跑 Agent / 高并发短请求？
  │   └─ 是 → SGLang（RadixAttention 在多轮对话和 Agent 场景下缓存命中率远超其他框架）
  │
  ├─ 需要最快的模型上线速度 + 最广模型支持？
  │   └─ 是 → vLLM（社区模型支持最广，新架构往往一周内就有 PR）
  │
  └─ 深度使用 HuggingFace 生态 + 需要 TGI 特有功能（如 watermarking）？
      └─ 是 → TGI
```

**默认推荐**：从 vLLM 起步。它是社区事实标准，文档最全、坑最少、部署最简单。等遇到明确瓶颈（吞吐不够、缓存命中率低）再评估 SGLang。

## 3. 三个关键场景的实测数据（量级参考）

> [!WARNING]
> 以下数字是 2026 年中业界报告的**量级**，不是精确 benchmark。实际跑你自己的 workload 测。

### 3.1 短对话（<2k context, 50-200 token 输出）

| 框架 | 相对吞吐 | 备注 |
|---|---|---|
| vLLM | 1.0× (基线) | |
| SGLang | 1.1-1.3× | RadixAttention 在短对话里优势有限 |
| TensorRT-LLM | 1.2-1.5× | 编译优化的效果在短序列上最明显 |

### 3.2 长文档问答（32k-128k context）

| 框架 | 相对吞吐 | 备注 |
|---|---|---|
| vLLM | 1.0× (基线) | Prefix caching 可用但不如 SGLang 高效 |
| SGLang | 1.5-2.5× | RadixAttention 前缀共享在这个场景**碾压** |
| TensorRT-LLM | 1.3-1.8× | 需要足够显存（通常 ≥4×H100） |

### 3.3 Agent 多轮（5-20 turns, 上下文逐步膨胀）

| 框架 | 相对吞吐 | 备注 |
|---|---|---|
| vLLM | 1.0× (基线) | 每轮重算 prefix 开销大 |
| SGLang | 2.0-4.0× | RadixAttention 自动共享各轮的前缀，**Agent 场景首选** |
| TGI | 0.8-1.0× | 无明显优化 |

## 4. SRE 的迁移检查清单

当你从一个框架迁到另一个时：

### 从 vLLM → SGLang

- [ ] 确认你的模型架构在 SGLang 支持列表中
- [ ] 对比同等硬件下的吞吐（用你自己的 workload profile 测，不要信公开 benchmark）
- [ ] 监控 RadixAttention cache hit rate（决定实际收益）
- [ ] 检查 structured output 行为是否一致（JSON schema 约束在框架间可能有微小差异）
- [ ] 确认 LoRA adapter 加载机制（如果你用 LoRA）

### 从任何框架 → TensorRT-LLM

- [ ] 确认模型能通过 ONNX 导出或直接 TRT 编译
- [ ] **预留 1-2 周模型上线周期**（编译 + 验证）
- [ ] 做 bf16/fp32 数值一致性回归测试（编译优化可能引入微小漂移）
- [ ] 确认团队有 CUDA/TensorRT 排障能力
- [ ] 评估量化版本的精度损失（TRT 的 FP8/INT4 实现与其他框架可能有差异）

---

## 5. 给 SRE 的一句话总结

> [!IMPORTANT]
> 四个框架的共同趋势是 **prefix caching 成为核心竞争力**——谁的缓存命中率高，谁的吞吐就翻倍。SGLang 的 RadixAttention 目前在这一点上领先。
>
> 但 SRE 选框架不能只看吞吐。**上线速度、模型支持广度、团队能力匹配度、故障恢复能力**同等重要。默认从 vLLM 起步，遇到 Agent 高并发瓶颈再评估 SGLang，追求极致性能且团队有 CUDA 能力再考虑 TensorRT-LLM。
