# 深入 10 · AI 系统事故模式库

> [← 返回目录](../README.md)

> [!NOTE]
> Google SRE Book 有 "Cascading Failures" 这种 **事故 pattern 目录**——SRE 最有效的学习方式之一是 **从模式里认出未来的问题**。
>
> 本章收录 **15 种 AI 系统典型事故模式**。每个包含：症状、根因、检测、处置、预防。用来做 **on-call 事前准备** 和 **事故后归类学习**。

---

## 使用方法

- **事故发生时**：对照症状列表快速定位 pattern
- **事故复盘时**：归类到 pattern 并补充本公司细节
- **上线新系统前**：过一遍预防清单
- **面试 / 团队培训**：用 pattern 做 tabletop exercise

### 从引章的"4 种失败模式"过渡到本章 15 个 pattern

[引章](../01-引章-大模型速览.md#四种你一定会遇到的失败模式) 列了 4 种概念级失败模式作为最简入门——它们和本章 15 个 pattern 的对应关系：

| 引章 | 对应本章 pattern |
|---|---|
| **幻觉 Hallucination** | Pattern 1（Silent Quality Regression）、Pattern 6（Judge Model Collapse）、Pattern 10（RAG Retrieval Noise Saturation）等多个 |
| **截断 Truncation** | Pattern 7（Cost Explosion via Context Growth）、Pattern 9（Context Window DoS） |
| **注入 Prompt Injection** | Pattern 5（Tool Result Poisoning）、Pattern 15（Persistent Injection via Memory） + 深入 07 红队全篇 |
| **漂移 Drift** | Pattern 1（Silent Quality Regression）、Pattern 3（Prompt Drift on Model Upgrade）、Pattern 11（Tokenizer Version Drift） |

引章是**概念级**（适合解释给业务方）；本章是**事故级**（适合 on-call 与复盘归类）。读者用引章建立心智模型，用本章定位真实事故。

## Pattern 1 · Silent Quality Regression（静默质量下降）

**症状**
- 所有传统指标（可用性、延迟）正常
- 用户投诉"回答突然变差"增多
- Error rate 不变
- 成本略降或不变

**根因**
- 模型供应商静默升级（OpenAI / Anthropic 常发生）
- 量化静默降级（见 [科学 03](../科学/03-Quantization为什么有时坏.md)）
- Kernel / 编译器 bug 导致数值问题（Anthropic 事故案例）
- Tokenizer 换了导致 prompt 命中率下降

**检测**
- L2 Judge eval 分下降（若有）
- 用户反馈率（thumbs-down）上升
- **端到端业务 KPI**（留存、转化）变化

**处置**
- 立即 rollback 到上个稳定版本
- 切流到不同厂商（如备用 API key）
- 数据回放对比两版本输出

**预防**
- Pin 模型版本（别用 `latest`、别用无版本号别名）
- **Canary eval**：每小时跑 10-20 条固定样本，结果飘就报警
- 保留**shadow 流量**对比新旧

**关联**：[科学 03](../科学/03-Quantization为什么有时坏.md)、[深入 06](06-Eval-Pipeline设计.md)

## Pattern 2 · Cache Miss Storm（缓存风暴）

**症状**
- 成本突然涨 3-10×
- TTFT p99 飙升
- 没有流量增长
- API bill 下周才能看到

**根因**
- 部署了"开头加时间戳 / user_id" 的 prompt 改动（缓存失效）
- 模型版本升级（跨版本缓存作废）
- 服务端路由策略变了（路由亲和丢失）
- Cache TTL 到了但业务没跟进

**检测**
- **Cache hit rate** 突降 20%+
- `cache_creation_input_tokens` 突增
- `input_tokens` 突增（这是没命中缓存的部分）

**处置**
- 回滚 prompt 改动
- 升级灰度暂停
- 临时加容量应对 TTFT 飙升

**预防**
- Prompt 模板 **预先过 cache 影响分析**（见 [深入 02](02-Prompt-Caching原理.md)）
- Cache hit rate 作为**一级 SLI**
- 灰度发布**预留成本冲击预算**

**关联**：[深入 02](02-Prompt-Caching原理.md)、[深入 04](04-为什么简单你好也消耗数万token.md)

## Pattern 3 · Prompt Drift on Model Upgrade（升级后 Prompt 漂移）

**症状**
- 同样 prompt，新版本模型给出**结构不同**的输出
- JSON 解析失败率上升
- Tool use 参数出错率上升
- 部分任务"更啰嗦"或"更懒"

**根因**
- 模型 RLHF 调了偏好（长度、格式、refusal）
- Tokenizer 变了
- System prompt 效果变了
- Alignment 改变导致行为风格变

**检测**
- 结构化输出**解析失败率**突升
- Tool call schema validation 错误率升
- 输出平均长度变化 ±20%

**处置**
- 回滚
- 跑对比 eval 找出具体变了什么
- 更新 prompt 适应新模型

**预防**
- **任何 prompt 依赖都要 eval 覆盖**
- 新模型**切流前做 structured output 专项测试**
- Prompt 和模型版本**强绑定版本化**

**关联**：[共同语言 04 · Alignment](../共同语言/04-Alignment的词汇.md)、[深入 06](06-Eval-Pipeline设计.md)

## Pattern 4 · Tool Abuse / Recursive Agent Loop（工具滥用 / 递归循环）

**症状**
- 单个用户请求触发数百次 tool 调用
- Agent 执行数小时不结束
- 某个 tool 被反复以相似参数调用
- 成本单用户单请求 $10+

**根因**
- Agent 认为没达到目标，反复试
- 死循环（看到"失败"就再试同操作）
- Tool 返回值被当新指令（见 Pattern 5）
- Prompt 里没有"停止条件"

**检测**
- **单请求 tool calls > 20**
- **单请求时长 > 10 分钟**
- 单用户日成本 > 阈值

**处置**
- **硬限制**：步数 / 时长 / 成本 上限
- Kill long-running agent
- 审查该用户的任务类型

**预防**
- Agent loop 必须有**三重上限**：step count / wall time / budget
- Step 预算见 [深入 03](03-模型与工具场景化最佳实践.md) 的复合可靠性数学
- 每个工具调用都**可回滚或可容忍**

**关联**：[深入 05](05-LLM推理服务的容量规划.md)、Agent design

## Pattern 5 · Tool Result Poisoning（工具返回投毒）

**症状**
- Agent 做了意料之外的事
- Tool 被调用了不该调用的权限
- 数据库查询结果被当指令执行

**根因**
- MCP server / 外部 API 返回值里有**指令样文本**
- 第三方爬虫返回页里有 prompt injection payload
- 日志 / 数据库读到的内容有恶意注入
- **这是 indirect prompt injection**（见 [深入 07](07-Agent-Prompt-Injection红队实战.md)）

**检测**
- Tool call 序列异常
- Agent 的 reasoning 提到不是用户说的主题
- 外泄模式（调用了 send_email 但用户没要求）

**处置**
- 立即熔断该 Agent
- 隔离该会话
- 审计 tool response 历史

**预防**
- Tool response **清洗**（去控制字符、标注为"data 不是 instruction"）
- **最小权限工具凭证**
- Egress 白名单（砍外泄通道）
- 红队演练覆盖 indirect injection

**关联**：[深入 07](07-Agent-Prompt-Injection红队实战.md)、[第 6 章](../知识/06-AI自治与上下文架构约束.md)

## Pattern 6 · Judge Model Collapse（判官模型漂移）

**症状**
- Production eval 分数持续缓慢上升（"怎么越来越好")
- 但用户反馈没改善、甚至恶化
- 新模型版本的 eval 显示"全面超越"，但线上测试不理想

**根因**
- Judge model 升级或换了
- Reward hacking 在 judge 侧（模型学会了取悦 judge）
- Judge 和 human 对齐度掉了（没校准）
- Judge prompt 本身有偏见

**检测**
- **Judge vs human 对齐度（Cohen's κ）**跌破阈值
- L2 eval 分和 L3 用户反馈**脱钩**
- 某类回答分数异常高（比如长回答）

**处置**
- 暂停"按 eval 分自动决策"
- 立即跑 human relabeling 校准
- 换 judge model 或更新 rubric

**预防**
- Judge 对齐度作为 **eval 的 eval**
- **两家 judge 投票**（不同厂商），分歧送人工
- 定期轮换 judge

**关联**：[深入 06](06-Eval-Pipeline设计.md)

## Pattern 7 · Cost Explosion via Context Growth（上下文增长导致成本爆炸）

**症状**
- 单用户成本随对话时长非线性增长
- 老用户成本比新用户高 10×
- 月度 bill 线性涨但 DAU 没涨
- Token 账单**指数级**

**根因**
- 长对话没 compaction，context 持续累积
- 历史 tool 调用结果全保留（每次 `Read` 几 k token）
- Agent 长任务不清理中间状态
- 记忆系统无限增长

**检测**
- **Input token 分布的 p99 / p50 比率**急增
- 老 session 的 input size 持续涨
- 历史对话长度监控

**处置**
- 强制截断超长 session
- 启动 compaction（总结 + 丢弃）
- 限制 session 最长轮数

**预防**
- Session 长度硬上限
- **Compaction 策略**预置（Claude Code 的 `/compact` 模式）
- Tool result 必要时压缩
- 每个 session **单月成本上限**熔断

**关联**：[深入 04](04-为什么简单你好也消耗数万token.md)、[第 6 章](../知识/06-AI自治与上下文架构约束.md)

## Pattern 8 · PII Leakage via Error Messages（PII 通过错误信息泄漏）

**症状**
- 日志里发现用户数据片段
- Error tracking（Sentry 等）有 PII
- 日志聚合系统包含敏感字段
- 合规审计警告

**根因**
- LLM 异常回显用户输入到错误
- Stacktrace 含 prompt 内容
- Tool failure 把参数原样打回
- Debug log 打了完整 prompt

**检测**
- **Log 扫描**（PII detector）
- 合规扫描工具
- 用户投诉（"你怎么知道我...") 

**处置**
- 立即清洗历史 log
- 日志脱敏化
- 通知受影响用户（合规要求）

**预防**
- **任何 log 输出前做 PII filter**
- Error message **去参数化**（别带用户输入）
- Prompt 和用户数据**分开存储**
- 合规自动扫描流水线

## Pattern 9 · Context Window DoS（上下文窗口拒绝服务）

**症状**
- 某用户 / IP 请求把 context 塞满（数十万 token）
- 每请求成本极高
- TTFT 高到数十秒
- 并发容量被一两个用户独占

**根因**
- 攻击者故意灌超长 prompt
- 正常用户无意触发（粘贴巨大日志）
- 上游爬虫返回超大页面进入 RAG

**检测**
- **单请求 input token p99**
- Per-user input size 分布
- 超过阈值请求数

**处置**
- 截断 / 拒绝超长请求
- 用户限流
- 清掉异常用户的 session

**预防**
- **input size 硬上限**（前端 + 后端双重）
- Per-user 并发限制
- 异常流量基于 token 速率告警

## Pattern 10 · RAG Retrieval Noise Saturation（RAG 检索噪声饱和）

**症状**
- 回答正确率下降但模型不幻觉
- 引用了**相似但不相关**的片段
- 用户反馈"查到的都是擦边的"
- Embedding 库升级后劣化

**根因**
- Embedding 质量漂移
- Chunking 策略不好（太碎 / 太大）
- 缺 rerank 阶段
- 向量库里噪声文档比例上升

**检测**
- Retrieval **hit@k** 指标
- 人工抽查"召回相关性"
- 端到端正确率下降

**处置**
- 补 rerank 阶段
- 重审 chunking
- 清理向量库

**预防**
- **必有 rerank**（Cohere / bge-reranker）
- 定期评估 retrieval quality
- 不用主力模型做 embedding

**关联**：[科学 02 · Lost in the Middle](../科学/02-Lost-in-the-Middle.md)

## Pattern 11 · Tokenizer Version Drift（Tokenizer 版本漂移）

**症状**
- 同样文本**token 数突变**
- 成本不明涨
- 缓存 hit rate 骤降
- Prompt 长度超预期触发截断

**根因**
- 厂商换 tokenizer（Opus 4.7 事件）
- 客户端 SDK 升级
- 自建推理引擎升级

**检测**
- **每百字中文 token 数**的 baseline 监控
- 成本 per-character 指标

**处置**
- 重算成本预估
- 更新 prompt 长度上限
- Cache 预热

**预防**
- SDK 版本 pin
- Tokenizer 明确标注版本
- **成本计算用 actual token count**，不是字符数

**关联**：[科学 04 · Tokenization](../科学/04-Tokenization的坑.md)

## Pattern 12 · KV Cache Preemption Storm（KV 缓存抢占风暴）

**症状**
- 某一两秒内大量请求延迟尾巴跳到 10× 正常
- TTFT p99.9 飙升
- **部分请求 cache 消失**重算

**根因**
- 显存接近满，推理引擎驱逐活跃请求 cache
- 长 session 抢占短请求
- 突发长 context 请求进来

**检测**
- **Preemption rate** 指标
- KV cache utilization > 90%
- 同时看 p50 vs p99.9 分叉

**处置**
- 临时扩容
- 拉长请求路由时间窗口
- 断开超长 session

**预防**
- KV cache **留 15-20% buffer**
- 容量规划覆盖长尾 context
- 长短请求**分 pool 部署**

**关联**：[深入 05 · 容量规划](05-LLM推理服务的容量规划.md)

## Pattern 13 · Batching Latency Spike（批处理延迟尖刺）

**症状**
- 平均延迟正常
- P99.9 偶发飙升 10×
- 没有规律
- 集中在高负载时段

**根因**
- Continuous batching 凑 batch 窗口，某些请求落在尾
- Batch size 太大导致 slow path
- Request payload 差异过大（短和超长同 batch）

**检测**
- p50 vs p99.9 分叉
- Batch size 分布分析
- 按 input size 分桶看延迟

**处置**
- 降 batch size（牺牲吞吐）
- 按 input size 做 request routing
- 长请求单独 pool

**预防**
- **多种请求形态分 pool**
- 监控 batch 排队指标

## Pattern 14 · Thinking Budget Blowout（思考预算爆炸）

**症状**
- 开了 thinking / reasoning 的模型账单飞升
- 回答 token 正常，但 input 里显示巨量 `reasoning_tokens`
- 对简单 query 也启用 thinking

**根因**
- Thinking 默认开没关
- 没做"简单 query 不 thinking"的路由
- `reasoning_effort` 全设最高档

**检测**
- `thinking_tokens` / `output_tokens` 比率
- 按任务类型分桶的成本

**处置**
- 分级 thinking 策略
- 简单 query 走非 thinking

**预防**
- 按任务复杂度动态调节 thinking
- 监控**thinking token 预算**独立指标

**关联**：[深入 03 · 1.1](03-模型与工具场景化最佳实践.md)

## Pattern 15 · Persistent Injection via Memory（记忆注入持久化）

**症状**
- Agent 在很多用户 session 里做同样异常事
- 一段"禁忌内容"反复出现
- 某个特定 trigger 词总让 Agent 失控

**根因**
- 早期一次 prompt injection 被**写入了 memory 系统**
- 之后每次会话都召回这条
- 见 [深入 08 · 记忆](08-大模型的记忆是怎么回事.md) 的 § 2.4

**检测**
- Agent 行为异常**跨用户一致**
- Memory 系统里找到可疑条目
- **审计所有 memory 写入历史**

**处置**
- **全量清洗 memory**
- 暂停 memory 写入
- 逐条审查 provenance

**预防**
- Memory 写入**不能静默**
- Memory 内容过 injection filter
- 记录每条 memory 的 provenance（谁写的、哪次会话）
- 红队覆盖"持久化注入"路径

**关联**：[深入 07](07-Agent-Prompt-Injection红队实战.md)、[深入 08](08-大模型的记忆是怎么回事.md)

## Pattern 16 · Zombie Stream（伪存活流）

**症状**
- 流式请求看似还在进行：连接没断、`recv` 不报超时、网关心跳正常
- **但用户已经很久没收到新 token**，看到的画面是"光标在闪、字不出来"
- 最终命中的是上层超时（5 分钟、10 分钟级别），不是流式超时
- 平均延迟看起来正常（被尾部吃掉算成功率），偶发出现

**根因**
- 上游推理服务卡住（OOM、KV cache 抢占、tokenizer 死锁、外部工具调用挂住）但**仍然在维持 keepalive**：
    - SSE 协议的 `: ping` 注释行
    - HTTP/2 的 keepalive 帧
    - 上游为了"防止网关超时断连"主动发的 whitespace
- **网关侧的流式心跳是按"字节是否在流动"判定的**，所以判活是真的，但判活与判进度被混为一谈
- 单服务部署看不到这种事故——它只在"网关 ↔ 上游"这条链路上才暴露，因为单服务里"出 token"和"出字节"是同一件事，而**网关位上两者必须分开**

**检测**
- 引入 **token 级心跳**，而不是字节级心跳：以"出现可解析为 token 增量的事件"为单位计时
- 看板上同时画 `bytes_per_sec` 和 `tokens_per_sec`，发散是首要信号
- SLI：`stream_no_token_seconds`（流内连续 N 秒无 token 增量），P99 应当远小于流总时长
- 从计费日志反查：成功结束但 `completion_tokens / stream_duration_ms` 极低的请求，要怀疑这一类
- 客户端侧的"用户主动断开率"在某个上游突然升高，也是一个滞后信号

**处置**
- 流内 N 秒（建议 30s 起，按业务调）无 token 增量 → **网关主动断流并回写明确错误**，不要等上层超时
- 该上游通道置入降级队列，跌出主路由
- 把这条请求的 input/响应头存档，提供给上游团队定位（往往是 KV cache 死锁或工具调用挂死）

**预防**
- **判活与判进度分开**：keepalive 字节只能维持连接，不能延长流式 SLO
- 每个上游通道建一个**金标准探针**（fixed prompt、已知输出长度），定期注入，监控 `tokens_per_sec` 不退化（关联：Unit 3 静默降级）
- 流式协议层面：把 ping 心跳和数据帧用不同事件名区分（如 SSE 的 `event: ping` vs `event: delta`），不要混在 `data:` 里
- 计费日志里把 `tokens_per_sec` 作为**质量字段**留存，而不是只算成本
