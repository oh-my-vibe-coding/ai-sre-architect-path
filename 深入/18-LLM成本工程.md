# 深入 18 · LLM 成本工程

> [← 返回目录](../README.md)  ·  相关：[深入 02 · Prompt Caching](02-Prompt-Caching原理.md)  ·  [深入 04 · Token 消耗](04-为什么简单你好也消耗数万token.md)  ·  [深入 05 · 容量规划](05-LLM推理服务的容量规划.md)  ·  [深入 13 · 离线批量推理](13-离线批量推理.md)

> [!NOTE]
> SRE 最常被追问的一句话："这个 Agent 为什么一个月花了 5 万美元？" 这一章给你回答这个问题的完整框架——从估算、归因到优化，每一步都可落地。

---

## 0. 为什么成本工程是 SRE 的事

很多组织把 LLM 成本归给"财务"或"产品"，但实际能回答"钱花哪了"的人只有 SRE。原因很简单：

- 成本不是由定价决定的，是由**调用模式**决定的
- 调用模式由**架构设计**决定——系统提示多长、缓存命中多少、Agent 走几步
- 这些全部是 SRE 的管辖范围

> [!IMPORTANT]
> LLM 成本是按 token 计费的，但 token 消耗是**工程行为的函数**。成本工程本质上是对调用行为的工程约束——它不是财务问题，是架构问题。

## 1. Token 成本估算公式

### 1.1 基础公式

一次 API 调用的成本：

```
单次成本 = (prompt_tokens × prompt_单价 + completion_tokens × completion_单价) / 1M
```

加上缓存折扣：

```
实际 prompt 成本 = cached_tokens × cached_单价 + uncached_tokens × 标准单价
```

### 1.2 2026 年中的量级参考

| 模型档位 | Prompt 单价 (per 1M token) | Completion 单价 | Cached Prompt 单价 |
|---|---|---|---|
| Frontier (Opus/旗舰) | $5-15 | $25-75 | $0.50-1.50 |
| 主力 (Sonnet/Flash) | $1-3 | $8-15 | $0.10-0.30 |
| 轻量 (Haiku/Nano) | $0.10-0.50 | $0.40-2.00 | $0.01-0.05 |

> [!WARNING]
> 上表是量级，不是报价。具体数字以厂商官网为准。这里记**量级**——主力模型的 prompt 比 completion 便宜 5-8 倍，cache 比标准 prompt 便宜 10 倍——这些比例关系比具体数字更耐用。

### 1.3 月成本的快速估算

```
月成本 ≈ 日均请求数 × 平均 prompt token × prompt 单价 / 1M
        + 日均请求数 × 平均 completion token × completion 单价 / 1M
        × 30
```

**经验口诀**：
- 一个日均 10 万次调用、每次 5k prompt + 500 completion 的 Sonnet 级应用 → **月成本约 $3k-8k**
- 加上 Agent 多步调用（×3-10 步）→ **$10k-80k**
- 加上 128k context 的长文档场景 → **$50k-200k+**

## 2. 成本归因：钱花在哪了

### 2.1 四层归因模型

```
总成本
  ├─ 模型层：哪个模型花了多少
  ├─ 功能层：哪个场景/能力花了多少
  ├─ 用户层：哪些用户/租户花得多
  └─ 结构层：system prompt / tool def / context / history 各占多少
```

**能做到第几层，取决于你在哪个位置**：

| 位置 | 模型层 | 功能层 | 用户层 | 结构层 |
|---|---|---|---|---|
| API 网关 | ✅ | ✅ (按 route/key) | ✅ | ✅ (唯一能看到全量的位置) |
| 推理服务 | ✅ | ❌ | ❌ | 部分 |
| 应用日志 | ✅ | ✅ | ✅ | 部分 (取决于日志打得多细) |
| 厂商账单 | ✅ | ❌ | ❌ | ❌ |

> [!TIP]
> **网关是实现四层归因的唯一位置**。这也是为什么 [深入 17 · §6](17-LLM网关的SRE视角.md#6-data-flywheel-在网关位的廉价起点) 说"计费日志是最好的飞轮起点"——它天然带着归因所需的所有字段。

### 2.2 结构层归因：一次调用的 token 账单解剖

以 Claude Code 类 Agent 的一次典型调用为例：

| 组成部分 | Token 数 | 占比 |
|---|---|---|
| System prompt（角色 + 规则 + 工具定义） | 3,000-8,000 | 30-50% |
| 项目上下文（CLAUDE.md / 文件内容） | 5,000-50,000 | 20-40% |
| 对话历史 | 2,000-80,000 | 10-30% |
| 当前用户消息 | 50-500 | <1% |
| **合计 prompt** | **10,000-140,000** | — |
| Completion（Agent 思考 + 回复 + 工具调用） | 500-5,000 | — |

**核心洞察**：用户实际输入（"你好" / "改个 bug"）占 token 消耗的 < 1%。这就是 [深入 04](04-为什么简单你好也消耗数万token.md) 的量化的账单。

### 2.3 成本异常的根因定位

当你看到月账单暴涨，排查顺序：

1. **先看模型层** — 哪个模型成本涨了？是不是有人把 Haiku 换成了 Opus？
2. **再看结构层** — prompt token 均值涨了？是不是有人在系统提示里加了长篇内容？
3. **再看功能层** — 哪个 API key / route 成本涨了？是不是某个场景上了 Agent 多步？
4. **最后看用户层** — 是不是某个租户用量突增？

> 90% 的成本异常在第二步就能定位。

## 3. Agent 链路的隐性成本

### 3.1 Agent 的成本放大效应

单次问答 vs Agent 的 token 消耗不在一个量级：

| 场景 | 典型 token 消耗/次 | 相对比例 |
|---|---|---|
| 简单问答（单轮） | 1k-5k | 1× |
| RAG 问答（检索 + 生成） | 5k-20k | 3-5× |
| Agent 单工具调用（search → read → answer） | 10k-40k | 5-10× |
| Agent 多步推理（SWE-bench 级） | 100k-1M+ | 50-200× |
| 长链 Agent 循环（工具调用 20+ 步） | 500k-5M+ | 200-1000× |

**Agent 的成本放大器有三个来源**：

1. **步数放大**：每个 tool call 往返一次 API，带着越来越长的上下文
2. **上下文膨胀**：每步的结果追加到历史，对话越长越贵（见 [深入 04](04-为什么简单你好也消耗数万token.md)）
3. **思考链消耗**：reasoning model 的 thinking token 不计入 completion 输出，但**计费**（Claude/OpenAI 的 extended thinking 都收费）

### 3.2 Agent 成本控制清单

| 控制点 | 做法 | 节省幅度 |
|---|---|---|
| 步数上限 | 硬限制单次任务最多 N 步（N ≤ 20） | 截断极端情况 |
| 上下文压缩 | 在 N 轮后触发摘要压缩历史 | 30-50% |
| 工具返回裁剪 | tool result 只保留必要字段，不返回全文 | 10-30% |
| 模型降级路由 | 简单意图用轻量模型，复杂的再走主力 | 40-70% |
| 缓存设计 | 系统提示 + 工具定义放最前，固定不变 | 50-90%（prompt 部分） |
| NOTES.md 模式 | 中间状态落盘，reset context 后 re-read | 20-40%（长链场景） |

## 4. 多模型路由的成本优化

### 4.1 路由策略的三种模式

**模式 A · 意图路由（Triage Router）**

```
用户请求 → 意图分类（轻量模型）→ 
  ├─ 简单问答 → Haiku/Nano ($)
  ├─ 代码/推理 → Sonnet/主力 ($$)
  └─ 复杂多步 → Opus/旗舰 ($$$)
```

- 意图分类本身消耗 ~100-500 token（成本可忽略）
- 如果 70% 的请求被路由到轻量模型，总成本降 40-60%
- **关键**：意图分类器必须比被路由的模型便宜一个数量级

**模式 B · 级联路由（Cascading Router）**

```
请求 → 轻量模型先答 → 
  ├─ 自信度高 → 直接返回
  └─ 自信度低 → 升级到主力模型重答
```

- 适合"多数简单、少数难"的场景
- 比意图路由多一层级联调用开销（约 +10-20% 成本），但准确率更高

**模式 C · 静态分桶**

```
按 API key / workspace 分 →
  ├─ 内部工具 → 轻量模型
  ├─ 对外产品 → 主力模型
  └─ VIP 客户 → 旗舰模型
```

最简单、最可控，但做不到请求级的精细路由。

### 4.2 路由的成本效益计算

假设：
- 轻量模型成本 = 0.1× 主力
- 意图分类成本 = 0.01× 主力
- 70% 的请求被正确路由到轻量

```
新成本 = 70% × 0.1 + 30% × 1.0 + 1.0 × 0.01 = 0.38× 原成本
节省 = 62%
```

**决定路由收益的唯一变量是"能被降级到轻量模型的比例"**。这个比例取决于你的请求分布——去查网关日志。

## 5. 缓存的经济学

### 5.1 缓存命中的成本模型

```
无缓存：prompt 全价 × prompt_tokens
有缓存（命中率 H，命中 prefix 长度 L）：
  缓存部分：cached_price × L
  未命中部分：full_price × (prompt_tokens - L)
  
节省 = H × L/prompt_tokens × (1 - cached_price/full_price)
```

**示例**（主力模型，cached_price = 0.1× full_price）：

| 命中率 | 命中长度占比 | 节省 |
|---|---|---|
| 80% | 60% | 80% × 60% × 0.9 = **43%** |
| 90% | 80% | 90% × 80% × 0.9 = **65%** |
| 95% | 90% | 95% × 90% × 0.9 = **77%** |

### 5.2 缓存设计的铁律

从 [深入 02](02-Prompt-Caching原理.md) 的机制推导出的成本规则：

1. **不变的部分放最前**：系统提示 + 工具定义固定 → 缓存命中率最高
2. **变化的部分放最后**：用户消息、检索结果 → 不影响前缀匹配
3. **不要在缓存前缀里放时间戳/user_id** → 每次都是不同的 token 序列
4. **模型版本升级 = 缓存归零** → 灰度期留出"缓存冷启动"成本预算（+50-100% 持续 5-10 分钟）

### 5.3 缓存命中率的监控

| 指标 | 计算方法 | 告警阈值 |
|---|---|---|
| Cache hit rate | `cached_tokens / (cached + uncached)` | < 70% |
| Cache creation rate | 新写入缓存数 / 分钟 | 突增 → prefix 在变 |
| Cache cost savings | 按定价折算节省金额 | 趋势下降 → 排查 |
| Per-route hit rate | 按 API route 分桶 | 某 route 异常低 → 该 route 的 prompt 设计有问题 |

## 6. 离线批处理的成本优势

[深入 13](../深入/13-离线批量推理.md) 详细讲了批处理。这里只讲成本账：

| 维度 | 在线 API | Batch API |
|---|---|---|
| Prompt 单价 | 标准价 | **5 折** |
| Completion 单价 | 标准价 | **5 折** |
| 缓存可设计性 | 取决于自然重复 | **可主动排序**凑前缀复用 |
| 资源类型 | On-demand GPU | **Spot/Preemptible** |
| 综合折扣 | — | **通常 60-80% off** |

**实操公式**：

```
批处理月成本 ≈ 在线月成本 × (1 - 批处理占比 × 0.6)
```

如果一个组织 70% 的 token 消耗走批处理（离线 eval、数据标注、合成数据生成），实际账单是全部走在线 API 的：

```
1 - 70% × 0.6 = 58%
```

**省了将近一半。** 批处理占比是 SRE 可以推动的——它取决于你是否把离线任务从在线 API 迁移到 Batch API。

## 7. 成本监控 Dashboard 的最小集

### 7.1 必须有的指标

| 指标 | 粒度 | 用途 |
|---|---|---|
| 日均 token 消耗（prompt + completion）| 按模型 × 按 route | 趋势 |
| 日均成本（$） | 同上 | 预算对比 |
| 平均 prompt token / 请求 | 按 route | 上下文膨胀检测 |
| 平均 completion token / 请求 | 按 route | 输出啰嗦检测 |
| Cache hit rate | 按模型 | 缓存效果 |
| Agent 步数分布 | P50/P95/P99 | 异常步数告警 |
| Top 10 高消费用户/租户 | 按日 | 租户治理 |
| 批处理占比 | 按月 | 离线迁移进度 |

### 7.2 告警规则

| 条件 | 级别 | 动作 |
|---|---|---|
| 日成本 > 预算 × 1.3 | P1 | 排查四层归因 |
| 平均 prompt token 周环比 > 30% | P2 | 查是否有人改了 system prompt |
| Cache hit rate 日环比下降 > 20% | P2 | 查 prefix 变化 / 模型升级 |
| 单请求 token > 200k | P2 | 查是否为 Agent 死循环 |
| 单用户日成本 > $500 | P2 | 查是否为 abuse / bug |

## 8. 成本优化的优先级路径

从**投入产出比**排序：

```
1. Prompt Caching（几乎零成本，50-80% prompt 节省）
   ↓
2. 批处理迁移（离线任务走 Batch API，60%+ 节省）
   ↓
3. 模型降级路由（意图路由 → 轻量模型，40-60% 节省）
   ↓
4. 上下文裁剪（压缩历史 / 工具返回 / 项目上下文，20-40% 节省）
   ↓
5. Agent 步数限制（截断极端循环，止损型）
   ↓
6. 切换 tokenizer 友好的模型（中/日/韩语言场景，20-50% 节省）
```

**前 3 步做完，通常已经省了 60-80%。** 后 3 步是边际优化——只在前 3 步做完还没达标时启动。

## 9. 给 SRE 的一句话总结

> [!IMPORTANT]
> LLM 成本不是"厂商定价 × 用量"那么简单。它是**架构决策的财务投影**——系统提示写多长、缓存怎么排、Agent 走几步、批处理占比多少，每一项都在账单上有一行。
>
> SRE 的独特价值在于：**只有你能从 token 消耗反推架构问题，只有你能从网关日志定位成本根因**。
>
> 把成本监控放进你的 Dashboard，和延迟/错误率同级——它不是财务指标，是你的**架构健康度指标**。
