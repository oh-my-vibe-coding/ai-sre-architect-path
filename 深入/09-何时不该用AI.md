# 深入 09 · 何时不该用 AI

> [← 返回目录](../README.md)

> [!NOTE]
> 整本书讲的是"如何用好 AI"。**这一章讲反面：什么时候不该用**。
>
> 一个架构师的核心判断力之一就是 **"能识别这件事不该 AI 化"**。缺这层判断，整个团队会走在**把 AI 硬塞一切**的错误方向。

---

## 0. 为什么需要"反 AI 化"判断

### 常见病症

- 看见新 Agent 框架就想重写老服务
- 把 rule engine 全替换成 LLM
- 任何"智能化"需求都默认上 LLM
- 忽视已经跑了 10 年的确定性方案

### 架构师的责任

每个**"这件事 AI 能做"**的背后都有**"这件事该不该让 AI 做"**。后者比前者难得多。

> [!IMPORTANT]
> **能做 ≠ 应该做**。这是架构判断的第一性问题。

---

## 1. 七类不该让 AI 做的场景

### 场景 1 · 精确计算与强一致性

**不该用 AI**：
- 金额计算（转账、对账、税金）
- 计数（统计次数、数元素个数）
- 精确时间（日程冲突检测、时区转换）
- 强一致性状态（库存、配额）

**为什么**：
- LLM tokenization 让数字表示不稳定（见 [科学 04](../科学/04-Tokenization的坑.md)）
- 采样有随机性
- 数学能力有天花板
- **一次算错 = 事故**

**该用什么**：
- 普通代码 + 数据库
- LLM 如果要参与，**必须**通过 tool use 调用确定性函数（calculator、SQL），自己不算

### 场景 2 · 已有高效确定性方案

**不该用 AI**：
- `grep` 能搞定的事别让 Agent 搜
- 正则能抽的字段别让 LLM 抽（除非正则难写）
- SQL 能查的数据别让 Agent "智能理解"
- 已有的 alerting 规则能触发就别让 AI 判断

**为什么**：
- 确定性工具**快 1000×、便宜 1000×、可审计**
- LLM 给同样问题不同答案
- 生产事故后追溯"为什么这次 AI 判断错了"几乎无解

**判断标准**：
> 如果一个问题能用**100 行以内的规则**可靠解决，**别上 LLM**。

### 场景 3 · 高频简单任务（成本账算不过）

**不该用 AI**：
- 每秒百万级的日志分类（简单关键词能做）
- 每请求都要过的 validation（schema 校验）
- 毫秒级响应需求的网关判断

**为什么**：
- LLM 推理延迟 ~200ms-3s；确定性规则 ~1ms
- 成本差 4-5 个数量级
- 在高频场景上，**一点点 per-call 成本**累积起来是**天文数字**

**Worked example**：
- 100k QPS 的 API gateway，每次调用都让 LLM 判断 = **$500k+/天**
- 同样场景用规则引擎 = **几百美元/月基础设施**

### 场景 4 · 合规 / 审计要求可解释决策

**不该用 AI**：
- 信贷审批（反歧视要求）
- 医疗诊断结论（监管要求）
- 法律判决（可解释性要求）
- 人事晋升 / 解雇（劳动合规）

**为什么**：
- LLM 决策过程**本质不可解释**（即便给 reasoning trace，那也是生成的说辞）
- 监管可能**强制你出示决策依据**
- "因为 GPT 这么说的"不是合法依据

**该用什么**：
- 决策树 / 规则引擎 / 经典 ML 模型（可解释的）
- LLM 可以**辅助**（总结、提取），但**不能是决策者**

### 场景 5 · 强时延敏感场景

**不该用 AI**：
- 高频交易决策（微秒级）
- 实时游戏 AI 行为（毫秒级）
- 网络拥塞控制
- 实时控制系统（飞控、工业）

**为什么**：
- LLM 推理最快也是 **~100ms TTFT**（见 [深入 01](01-首包延迟与吞吐的影响因素.md)）
- 带 thinking 的更慢
- 某些场景延迟就是生死线

### 场景 6 · 小样本决策（模型不可靠的边际）

**不该用 AI**：
- 极罕见事件的分类（年度 1-2 次的故障）
- 高度专业的小众语言翻译
- 公司内部 jargon 的解析
- 未见过的新型数据格式

**为什么**：
- LLM 在长尾场景自信地犯错
- 小样本上**幻觉率极高**
- 你自己都缺数据训，模型更没见过

**该用什么**：
- 专家系统 + 规则
- 人工 in the loop
- 细分领域的专用 NLP 工具

### 场景 7 · 不可回滚 + 高 Blast Radius 的自动化

**不该用 AI**：
- 删除生产数据
- 自动执行不可逆的金融操作
- 改动基础设施配置
- 发送对外通信（邮件、短信）
- 自动部署到生产

**为什么**：
- Prompt injection 风险（见 [深入 07](07-Agent-Prompt-Injection红队实战.md)）
- 模型一时抽风
- 事故时追责困难
- 符合**致命三角**的"绝对禁止"档

**该用什么**：
- Human-in-the-loop（强制人类确认）
- AI 只能**建议**，不能执行

## 2. 八个可识别的"被 AI 诱惑"信号

看到以下任何一个，要**停下来质疑**：

| 信号 | 背后常见的问题 |
|---|---|
| "用 LLM 比较灵活" | 通常意味着你缺需求，不是工具不够 |
| "让 AI 自动处理这类情况就行" | "这类情况"没定义，边界模糊 |
| "我们先上 LLM，后面再优化" | 一旦上了就撤不下来，预算悄悄变成常态 |
| "Prompt 调一调就行" | 你其实在做"程序员工作"但用了最低效的表达 |
| "反正 API 也不贵" | 没算规模成本，小试算单价，上量算总价 |
| "AI 会越来越强" | 同样任务两年没解决，不是 AI 弱，是方案错 |
| "加个 LLM 看看效果" | 加法容易，减法难；上了就拿不下来 |
| "别人都在用 LLM 做这事" | FOMO 不是技术理由 |

## 3. 替代品清单（非完整，重要入门）

### 文本处理
- `grep` / `ripgrep` — 找字符串，不需要 LLM
- Regex — 结构化抽取的首选
- `awk` / `jq` — 管道处理
- sentence tokenizer + bag-of-words — 分类入门

### 判断 / 分类
- 规则引擎（Drools、open-policy-agent）
- 决策树（可解释）
- 经典 ML（逻辑回归 + feature engineering）
- 加权 scoring 系统

### 搜索 / 检索
- Elasticsearch / OpenSearch — 关键词就够
- PostgreSQL 全文搜索（`pg_trgm`）
- **只有语义搜索才上 embedding**

### 监控 / 告警
- Prometheus + AlertManager — 阈值告警
- 异常检测算法（3σ、IQR）
- 时序预测（ARIMA、Prophet）
- **不要用 LLM 替代**

### 总结 / 摘要（还没到 LLM 的场景）
- TextRank（无监督关键句抽取算法）
- TF-IDF 排序
- 模板化 + 填空

## 4. 降级决策树

遇到一个需求，先走这棵树：

```
需求来了
  │
  ├─ 能用 100 行以内规则解决？ ─── 是 ─► 用规则
  │                              否
  │                               │
  ├─ 有可用的经典 ML / 启发式？ ── 是 ─► 用它
  │                              否
  │                               │
  ├─ 场景规模每天 < 1000 次？ ──── 否 ─► 重新评估成本
  │                              是
  │                               │
  ├─ 后果可回滚？监管允许？ ────── 否 ─► 回到"该不该 AI"
  │                              是
  │                               │
  ├─ 人工监督可行？ ──────────────► 用 AI + HITL
  │                              否
  │                               │
  └─► 真正全自动 Agent（罕见，需要最严格的边界和红队）
```

**多数合理场景止步于前三步**。很多"一上来就做 Agent"的设计是跳过了前三步。

## 5. SRE 对 AI 的"反身性"评估框架

每个"我们要上 AI" 的提议，问四个问题：

### Q1 · **成本账**
全量规模下单月成本？比现有方案贵多少？**值不值得**？

### Q2 · **故障账**
AI 出一次错，业务损失多少？追责链条清晰吗？**能承受吗**？

### Q3 · **替代账**
如果 5 年后 AI 被监管禁用或变贵 10×，**有退路吗**？

### Q4 · **人才账**
这个系统 2 年后谁维护？需要**什么稀缺技能**？

**任一题答不清**，应该退回去重新设计，不是硬上。

## 6. 现实世界的"反模式"

### 反模式 A · "一切皆 Agent"
把 monorepo 的 CI 改成 Agent 来跑。结果：慢 10×、贵 100×、出错率高。**用 CI workflow 就行**。

### 反模式 B · "LLM 做 Schema 验证"
用户上传 JSON，让 LLM 检查字段合法。**JSON Schema 本身就是干这事的**，快、免费、确定。

### 反模式 C · "AI 写正则"
把"用户说想要的格式"交给 LLM 生成正则。**生成了就测，错了再生成** —— 你其实可以直接让工程师写。

### 反模式 D · "Agent 当 cron"
让 Agent 每小时跑一次检查。**crontab + python script 就够了**，不用"智能"。

### 反模式 E · "LLM 替代 alerting"
让 LLM 判断日志是否异常。**Prometheus 规则或简单 anomaly detection 更靠谱**。

### 反模式 F · "Everything-as-Chat"
把任何表单界面改成对话。多数用户**不想和产品聊天**，只想填表提交。

---

## 7. 健康的 AI 使用原则（正面对立面）

- **AI 擅长处理模糊** → 把"准确"的交回确定性工具
- **AI 擅长理解语言** → 但别让它"决定"
- **AI 擅长辅助** → 别让它"自主"，除非环境可控
- **AI 擅长探索** → 生产稳态还是经典方案
- **AI 擅长初稿** → 但你得是最终决策者

---

## 8. 给 SRE 的一句话总结

> [!IMPORTANT]
> 一个成熟的 SRE 架构师在 AI 时代的核心价值，**有一半是能说"这件事不该用 AI"**。
>
> 当团队走向"把一切 Agent 化"时，**挡在前面的人往往就是 SRE**——因为他们最清楚一次错误的事故代价。
>
> **"AI 可以做"和"应该让 AI 做"是两个问题。**
