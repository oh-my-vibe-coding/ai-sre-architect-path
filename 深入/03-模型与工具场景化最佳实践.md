### TL;DR（6 句话）

1. **推理 / Thinking 成为一线能力**：2026 年的"旗舰"必须附带 test-time compute（Claude Extended Thinking、OpenAI o-series、DeepSeek-R1、Gemini 3 Deep Think）。没有 thinking 模式的"纯指令模型"已经掉出 SWE-bench Top 10。[^o1][^r1][^ext]
2. **1M Context 标配，但"有效上下文"远小于宣称值**：RULER 基准显示"宣称 ≥32K 的模型只有一半能真正用到 32K"；Gemini 1.5 Pro 宣称 1M、实际有效 ~128K；Llama 3.1-70B 宣称 128K、实际 ~64K。[^ruler]
3. **中国系模型正在占领真实 token 用量**：OpenRouter 付费用量 Top 10 里 Kimi K2.6、DeepSeek V4 Flash、MiniMax M2.7、智谱 GLM-5.1、Tencent Hy3、Step 3.5 合计超过一半；MiniMax M2.5 与 Gemini 3 Flash 并列 SWE-bench Verified 第 2-3 位（75.80%）。[^openrouter][^swe]
4. **Coding / Agent CLI 已经分化为三派**：CLI 派（Claude Code / Codex CLI / Gemini CLI / opencode）、IDE 派（Cursor / Windsurf / Copilot）、可组合派（Aider / Cline / Continue）。Claude Code 以 126k+ stars 领先，但 opencode（anomalyco fork）以 164k+ stars 成为开源首选。[^cc][^oc]
5. **推理基建走向"解耦"（Disaggregated Serving）**：vLLM V1 引擎与 SGLang 0.5 都把 prefill/decode 解耦作为一等公民，SGLang 报告 GB200 上 decode 吞吐提升 2.7×；这是 SRE 架构师必须理解的新 dimension。[^vllm][^sg]
6. **Prompt Caching 是成本优化的最大杠杆**：Anthropic 读取折扣 ~90%、重用 2 次即回本；OpenAI 自动 caching 50% 折扣；DeepSeek cache-hit 与 cache-miss 价差达 **50×**（$0.0028 vs $0.14 per MTok）—— 不做缓存优化等于白烧钱。[^ds]

## 第一部分 · 技术流行趋势（2025-2026）

> **稳定部分**。这些趋势 1-2 年内不会推翻，只会深化。具体数字随时间漂移。

### 1.1 Scaling 的转向：从参数到"推理时计算"

**旧范式**：预训练投入越大 → 模型越强（Kaplan / Chinchilla scaling laws）。
**2024-2026 新范式**：**test-time compute** 成为第二条 scaling 曲线——**让模型在推理时"多想一会"**，质量随"思考 token 数"继续提升。

| 代表 | 机制 | 来源 |
|---|---|---|
| **OpenAI o1 / o3 / o4 series** | RL 训练的 chain-of-thought；训练时 + 推理时**双轴** scaling | 《Learning to Reason with LLMs》(2024-09)[^o1] |
| **DeepSeek-R1** | 纯 RL（无人工标注 reasoning traces）涌现自反思 / 自验证；可蒸馏到小模型 | arXiv 2501.12948 / Nature 2025[^r1] |
| **Claude Extended Thinking** | 每请求可配置"可见思考预算"，输出前有独立 thinking phase | Anthropic docs[^ext] |
| **Gemini 3 Deep Think** | Gemini 3.1 Pro 的"思考"模式 | Google AI Studio 2026 |

#### SRE 架构师必须内化的三件事

1. **Reasoning token 是单独计费的**（Anthropic thinking token / OpenAI reasoning token），**预算要乘以 3-10x**。
2. **Thinking 阶段没有流式输出**，UI 不要骗用户说"正在生成"——要展示"正在思考"。
3. **对延迟敏感的场景别用 thinking**；对质量敏感的场景（复杂推理、架构决策）必须用。

### 1.2 MoE 成为主流架构

**MoE（Mixture of Experts）**：模型参数量大，但每次推理只激活一小部分。好处是"大而不贵"。

2025-2026 主流 MoE 产品：

| 模型 | 总参 | 激活参 | 备注 |
|---|---|---|---|
| DeepSeek V3 / V4 | 685B / 1T+ | ~37B | 开源 MoE 标杆[^ds] |
| Llama 4 Maverick | 400B+ | ~17B | Meta 首个全 MoE 旗舰[^llama4] |
| Qwen 3.6-35B-A3B | 35B | 3B | 阿里 MoE 代表[^qwen] |
| Gemini / Claude / GPT-5 | 未公开 | - | 普遍推测为 MoE |

#### 对推理基建的含义

- **Expert Parallel (EP)** 成为部署 MoE 的第三种并行（除 TP/PP 外）
- **Routing imbalance** 是新故障模式（某些 expert 被过度路由 → 热点 GPU）
- **显存估算变复杂**：不是参数总量，而是"激活 + KV cache"

### 1.3 上下文长度的军备竞赛与"有效长度"幻觉

#### 宣称值 vs 实际可用

| 模型 | 宣称 | RULER / NIAH 实测有效 | 来源 |
|---|---|---|---|
| Gemini 1.5 Pro | 1M | **>128K**（好于平均）| RULER[^ruler] |
| Llama 3.1 70B | 128K | **~64K** | RULER[^ruler] |
| GPT-4 (1106) | 128K | **~64K** | RULER[^ruler] |
| Llama 4 Maverick | **10M** | 未见 RULER 公开结果 | [未验证] |
| Claude Sonnet 4.5 (1M 版) | 1M | 1M 版 SWE-bench 78.2% vs 200K 版 77.2%——**只差 1 个点**[^sonnet45] | Anthropic |

> [!IMPORTANT]
> **SRE 要知道**：RULER 核心发现——"宣称 ≥32K 的模型，只有**一半**能在 32K 长度下有效工作"[^ruler]。不要把"标称 context 大 = 能用"当真。

#### 实用推论

- **超过 128K 就要验证你的场景**。单个"needle in haystack"通过 ≠ 复杂多针任务通过。
- **Chunking + Rerank 永远优于裸喂长 context**，即使模型号称能接受。
- 长 context **推理成本线性涨**（prefill 成本 ~ 输入 token 数），1M 输入意味着数秒级 TTFT（见[深入 01](01-首包延迟与吞吐的影响因素.md)）。

### 1.4 Agentic 能力：从 "补全" 到 "长任务"

- **Long-running tasks**：Anthropic Project Vend 让 Claude Sonnet 3.7 运营自动贩卖机数周[^vend]；Claude Code / Codex CLI 能连续工作数小时
- **Computer use / Browser use**：OSWorld 基准（真实桌面任务）最佳模型仅 **12.24%**，人类 **72.36%**[^osworld]——**这个差距是 2026 年最大的未兑现机会**

### 1.6 推理基础设施的三个关键演化

1. **Disaggregated Serving（PD 解耦）成为一等公民**
   - Prefill 和 Decode 分到不同 GPU pool（前者需要算力，后者需要带宽）
   - vLLM V1 / SGLang 0.5 都已原生支持；SGLang 报告 GB200 上 decode 吞吐提升 **2.7×**[^sg]
2. **Prefix Caching 从可选变强制**
   - vLLM 的 Automatic Prefix Caching、SGLang 的 RadixAttention
   - 不开启就是白付钱（RAG / 长系统提示场景成本可降 80%+）
3. **Speculative Decoding 商品化**
   - EAGLE / DFlash / n-gram / suffix 多种方法集成
   - 单请求 tokens/s 可 2-4× 提升

**分水岭**：工具不再是"帮你打字"，而是"按权限给你干活"。SRE 架构师的新责任是**定义权限边界**（见[第 6 章](../知识/06-AI自治与上下文架构约束.md)）。

### 1.8 2025-2026 其他重要趋势（速扫）

- **多模态原生（Multimodal Native）**：Gemini 3.1、GPT-5.4、Claude Opus 4.7、Meta Muse Spark 都是从训练第一天就 multimodal；"文本+图像"不再是后接的 vision adapter[^muse]

- **Meta Superintelligence Labs（MSL）与 Muse Spark**：2026-04-08 发布，宣称 compute efficiency 10× Llama 4 Maverick，Humanity's Last Exam 58%、FrontierScience Research 38%[^muse]

## 第二部分 · 工程最佳实践

> **稳定部分**。场景-首选-关键考虑的三层结构，模型名可以换，结构不变。

### 2.1 选型总原则

> **不要被榜单牵着走。先定义场景，再列约束，再做你自己的 20 条样本 eval。**

三条铁律：
1. **多源对照**：LM Arena（主观）+ SWE-bench（硬指标）+ OpenRouter（用量）+ 你的小样本
2. **先廉后贵**：便宜模型能解决就别上旗舰；`Haiku 4.5 / Gemini Flash / DeepSeek V4 Flash` 覆盖 70% 日常任务
3. **对任务用 Thinking，对速度用非 Thinking**：同一家厂商的 thinking/non-thinking 版本是不同模型，不要混用

### 2.2 八大场景最佳实践

#### 场景 A · 代码生成与重构

| 子场景 | 首选 | 备选 | 自托管 |
|---|---|---|---|
| 大仓库 agentic 编码 | **Claude Opus 4.7 Thinking** / Sonnet 4.6 | GPT-5.5 / GPT-5.2 Codex | DeepSeek V4 Pro / Qwen3.6-35B |
| 日常补全 | Sonnet 4.6 / Gemini 3 Flash | GPT-5.4 mini | DeepSeek V4 Flash |
| 算法/竞赛难题 | Opus 4.7 Thinking / o4 / GPT-5.5 High | Grok 4.3 reasoning | Hermes 4-405B |

**关键考虑（不随模型变）**：
- **项目上下文文件** (`CLAUDE.md` / `AGENTS.md`) 是命中率的决定因素
- **Tool use 白名单**（只读命令默认；变更类动作强制 confirm）
- **Context 定期压缩**：长对话用 `/clear` 或 subagent 重启，比硬撑 1M 有效
- **双模型分层**：生成用贵模型，code review 用便宜模型

#### 场景 B · SRE 运维诊断

| 子任务 | 首选 | 理由 |
|---|---|---|
| 长日志 / trace 分析 | **Gemini 3.1 Pro**（1M 有效最好）/ Sonnet 4.6 | Context 长 + 便宜 |
| 结构化字段抽取（事件摘要）| Haiku 4.5 / Gemini Flash-Lite | 便宜、延迟低 |
| 事故根因推理 | Opus 4.7 Thinking / GPT-5.5 High | Thinking 值得多付 |
| 工单自动分类 | Haiku 4.5 / DeepSeek V4 Flash | 高吞吐批处理 |

**关键考虑**：
- **Tool use 必须白名单**（见[第 6 章](../知识/06-AI自治与上下文架构约束.md) 致命三角）
- **强制引用日志来源**（每个结论附原日志行号）
- **先结构化再推理**：让模型先抽取字段成 JSON → 再基于 JSON 推理（比直接"分析日志"准得多）
- **RAG 你的 runbook**：通用模型不知道你家环境，必须喂

#### 场景 C · RAG 与知识问答

| 子场景 | 首选 |
|---|---|
| 大文档单次总结（合同、报告）| Gemini 3.1 Pro（1M context + 单次便宜）|
| 中小文档 + 高质量答案 | Sonnet 4.6 |
| 高频问答（成本敏感）| Haiku 4.5 / Gemini 3 Flash-Lite |
| 自托管 | Qwen 3.6 / Llama 4 Scout |

**关键考虑**：
- **固定文档放前面** → prompt caching 命中，**成本降 80-90%**（见[深入 02](02-Prompt-Caching原理.md)）
- **强制引用来源**——对抗幻觉的底层护栏
- **Chunking 小而重叠**：500-800 token / 片，相邻重叠 10-20%
- **Embedding 用专用模型**（bge-m3 / E5 / Cohere Embed），不要用主力模型做 embedding
- **别省 Rerank**：检索 Top-50 → Rerank Top-5（bge-reranker / Cohere Rerank）

#### 场景 D · 结构化数据抽取

| 子场景 | 首选 |
|---|---|
| 高吞吐（日百万条）| Haiku 4.5 / Gemini Flash-Lite / DeepSeek V4 Flash |
| 复杂嵌套 schema | Sonnet 4.6 |
| 多语言（中英混合）| Qwen 3.6 |

**关键考虑**：
- **用厂商的 Structured Output / JSON Schema 强制约束**，不要靠 prompt 让它"输出 JSON"
- **Temperature = 0 或 0.1**
- **永远做 eval**：准确率目标（不是 100%）+ 漂移监控
- **失败样本自动回收做下一轮 eval 数据**（data flywheel，见[第 7 章](../知识/07-质量可观测性与DataFlywheel.md)）

#### 场景 E · 复杂推理 / 规划 / 数学

| 子场景 | 首选 |
|---|---|
| 多步推理 | **Claude Opus 4.7 Thinking** / OpenAI o4 / GPT-5.5 High |
| 数学 | 同上 + DeepSeek-R1 distilled |
| 代码推理（边想边写）| Opus 4.7 Thinking |

**关键考虑**：
- **开 extended thinking / reasoning mode**；接受 3-10x 成本
- **不要做流式 UI**——推理阶段用户看不到中间输出
- **Reasoning token 单独计费**，预算要分开算
- **Fallback 到非 thinking 版本**：对简单 query 自动降级

#### 场景 F · 视觉理解

| 子场景 | 首选 |
|---|---|
| 通用图像理解 | Gemini 3.1 Pro / GPT-5.4 / Claude Sonnet 4.6 vision |
| OCR | Gemini 3.1 Pro / Qwen 3.5 Omni |
| 截图理解（UI / 仪表盘）| Opus 4.7 / GPT-5.4 |

**关键考虑**：
- **图像分辨率决定上限**——太小会漏
- **先 describe 再 reason**：让模型先描述图像，再基于描述推理
- **多图累积成本**——每张图都占 token

#### 场景 G · 长任务 Agent

**首选**：Claude Opus 4.7（长任务专项优化）/ Sonnet 4.6
**备选**：GPT-5.4 + Codex CLI

**关键考虑（比模型选择更重要）**：
- **上下文工程**：compaction 策略、subagent 隔离、外部记忆 `NOTES.md` 模式
- **Checkpoint**：Agent crash 后能从中间继续
- **全审计日志**：所有 tool 调用必须可回放
- **预算上限**：token + 时长；跑飞了自动停

#### 场景 H · 本地 / 私有部署

| 规模 | 推荐 |
|---|---|
| 单 H100 / 消费卡 | Qwen 3.6-27B / Llama 4 Scout / Mistral Small 4 |
| 多卡 | Llama 4 Maverick / DeepSeek V4 Pro / Qwen 3.6-35B MoE / Hermes 4-405B |
| 边缘（M 系列笔记本 / iGPU）| Ministral 3 / Qwen 3.6-3B / llama.cpp GGUF 量化 |

**关键考虑**：
- **用 vLLM / SGLang**，不要裸跑 transformers
- **量化（INT8/INT4/FP8）前做 eval** 确认无质量回退（[第 5 章](../知识/05-AI推理服务的可靠性工程.md)静默降级）
- **Tool use / Structured output 支持度差异大**，先验证再选型
- **运维成本**往往比 GPU 成本更贵（人力 + 值班）

### 2.3 Prompt Caching 使用策略

**单一最强的成本优化手段**。[深入 02](02-Prompt-Caching原理.md) 详述原理，这里只讲最佳实践。

#### 三家对比（2026-05）

| 平台 | 触发 | 写成本 | 读成本 | TTL |
|---|---|---|---|---|
| Anthropic | 显式 `cache_control` | ~1.25× 常规 | ~0.1× 常规（-90%）| 5 分钟（可延长 1h beta）|
| OpenAI | 自动（≥1024 token）| 无额外 | ~0.5× 常规（-50%）| 5-60 分钟 |
| Google Gemini | 显式创建 cached content | 按存储时长付费 | ~0.25× 常规 | 用户指定 |
| **DeepSeek** | 自动 | 无额外 | **cache-hit $0.0028 / cache-miss $0.14 per MTok（差 50×）** | 未明确公布 |

[^ds]

#### 落地清单

- ✅ **固定内容放前面**：系统提示 → 参考文档 → 用户当前请求
- ✅ **用户 ID / 时间戳 / session ID 放后面或省略**
- ✅ **版本号变更 = 全体缓存失效**，灰度时预留成本冲击预算
- ✅ **路由亲和**（自建服务）：同 session 到同实例
- ✅ **监控 cache hit rate**：突降 20% 立即排查
- ❌ **别以为 "cache_control" 加了就一定命中**——监控真实命中率

### 2.4 成本优化四板斧

1. **分层**：70% 流量 → 便宜模型（Haiku / Flash-Lite / DeepSeek V4 Flash）；剩 30% → 旗舰
2. **Caching**：见 2.3，RAG / 长系统提示场景能省 80%+
3. **Batch API**：非实时场景（离线抽取、定期报告）→ **OpenAI / Anthropic Batch 折扣 50%**
4. **Speculative decoding**（自建服务时）：2-4× tokens/s 提升，成本相应下降

### 2.5 可观测性与 Eval 最佳实践

> **核心洞察（重复但必须反复强调）**：Traces ＝ Eval Dataset ＝ 生产监控。一套系统，不是两套。

#### 三层 Eval 体系

```
L1 Assertion（硬规则）
  ├─ 字段存在性 / 类型 / 长度
  ├─ 必要格式（JSON valid? URL 有效?）
  └─ 黑名单词（PII / 脏话 / 竞品名）

L2 Judge 模型评分
  ├─ 用 LLM 给输出打分（relevance / faithfulness / correctness）
  ├─ 必须定期校准：sample 100 条让人打分，计算 judge-human 一致性（Cohen's κ）
  └─ κ < 0.7 → judge 不可信，换 judge 模型或改 rubric

L3 A/B 测试 / 用户反馈
  ├─ 线上分流
  ├─ 收集 thumbs up/down、编辑距离、留存率
  └─ 权威最高，但样本慢
```

#### Evals 自身的 SLO

- **Eval 覆盖率**（生产样本有多少被 eval pipeline 评估）
- **Eval 延迟**（样本到 eval 结果的时间）
- **Judge-human 对齐度**（不对齐 = eval 失效）
- **Eval pipeline uptime**（evals 挂了你就瞎了）

### 2.6 安全与自治边界（极简提示）

完整内容见[第 6 章](../知识/06-AI自治与上下文架构约束.md)。**极简版**：

```
Lethal Trifecta（不受信输入 + 工具访问 + 外泄通道）
  → 在基础设施层砍掉一条腿（通常是"外泄通道"）
  → 沙箱 / egress 白名单 / capability-scoped 凭证
  → 禁止 Agent 修改自己的配置 / 凭证
```

### 2.7 工具使用反模式清单（**务必避免**）

- ❌ **用 prompt "请输出 JSON"**（要用 Structured Output / JSON Schema）
- ❌ **系统提示里放时间戳 / UUID**（缓存全部失效）
- ❌ **让 AI 操作生产 + 无白名单**（致命三角）
- ❌ **不做 eval 就上线**（你根本不知道它好不好）
- ❌ **把 Temperature 设为默认 1.0**（SRE 场景全错）
- ❌ **Streaming + Thinking 混用**（思考阶段无输出，用户以为卡死）
- ❌ **长 context 硬塞**（效果差且贵，不如 RAG）
- ❌ **用通用 embedding 模型做特定领域 RAG**（精度差）
- ❌ **所有请求用同一个旗舰模型**（70% 任务是过度配置）
- ❌ **Prompt caching 标记了就不管**（必须监控命中率）

### 4.7 2025-2026 新范式 / 易错过的演化

1. **Disaggregated Serving（PD 解耦）成为事实标准**：vLLM V1 + SGLang 0.5 都把 prefill/decode 分到不同 GPU pool；SGLang 报告 GB200 decode 吞吐提升 2.7×[^vllm][^sg]
2. **Skills / Hooks / Subagents 成为 CLI 工具一等公民**：Claude Code 形式化了这三个概念；Agent SDK 暴露给自建 Agent 用[^cc]
3. **反思式 Prompt 优化（GEPA）**：DSPy 新增，利用 LM 自反思迭代优化 prompt；agent 任务显著优于 MIPROv2[^dspy]
4. **Cognition 收 Windsurf + 专用 SWE 模型**：OpenAI 收购失败后 Cognition（Devin）入场；SWE-1.5 为 Workflow 专用模型[^win]
5. **Apple Silicon / AMD 本地推理拐点**：llama.cpp Metal + MLX 互操作，M 系列笔电跑 70B 量化；ROCm 6.x 让 AMD MI300/MI355 在 vLLM/SGLang 一等支持[^vllm][^sg]
6. **新编码基准**：SWE-bench Verified 仍是头条；**Terminal-Bench**（shell agent）、**SWE-bench Multimodal / Multilingual** 逐渐被 CLI 工具采用

## 第五部分 · 选型决策 7 步流程

> **稳定部分**。任何时候遇到选型问题，走这 7 步。

```
1. 定义场景
   └─ 对应第二部分哪个子场景（A-H）？
   └─ 如果跨多个场景，拆开分别决策

2. 列出硬约束
   ├─ 数据敏感度 → 是否允许 API / 托管？
   ├─ 延迟要求 → p99 TTFT < 多少？
   ├─ 成本上限 → 每百万 token / 每请求预算
   ├─ 合规 → 必须 VPC / 必须某地域？
   └─ 可控性 → 是否需要固定模型版本？

3. 列出软约束
   ├─ 生态（tool use / structured output 支持度）
   ├─ 文档 / SDK 质量
   └─ 厂商稳定性（API breaking change 历史）

4. 短名单 3 个
   ├─ 闭源旗舰 1（Claude / GPT / Gemini 旗舰之一）
   ├─ 性价比档 1（Haiku / Flash / DeepSeek V4 Flash 之一）
   └─ 开源备选 1（Qwen / Llama / Mistral 之一）

5. 做你自己的 20-50 条样本 Eval
   ├─ 从生产取真实样本（必要时脱敏）
   ├─ 定义 pass / fail 标准
   ├─ 测三方：准确率 + 延迟 p50/p99 + 成本
   └─ 对照 LM Arena / SWE-bench 看方向是否一致

6. 小流量上线
   ├─ 影子流量（双跑，不影响用户）一周
   ├─ 然后 5% 真实流量一周
   └─ 观察：cache hit rate、质量漂移、成本、失败率

7. 决定扩量 or 回退
   ├─ 扩量：逐步 25% → 50% → 100%
   ├─ 回退：分析失败案例，回到第 3 步
   └─ 每次都留可回滚开关（feature flag）
```

**一条常见错误**：直接看榜单 Top 1 就上线。**你自己场景的 20 条样本 eval，权重高于任何公开榜单**。
