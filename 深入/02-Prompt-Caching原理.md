# 深入 02 · Prompt Caching 原理：缓存是如何创建和命中的

> [← 返回目录](../README.md)  ·  对应知识章节：[第 5 章 · AI 推理服务的可靠性工程](../知识/05-AI推理服务的可靠性工程.md)  ·  深入推理底层时再读：[深入 01 · 首包延迟与吞吐](01-首包延迟与吞吐的影响因素.md)

这一节用一个比喻+一个具体场景把直觉建立起来，再讲机制和工程细节。**不需要你已经读过深入 01**——只在最后引用它的术语作为补充说明。

## 0. 一个比喻：每天对失忆学生重讲开场

想象你每天给同一个学生上 1 小时课。这个学生**每天都失忆**：每次见面，他都不记得你是谁、上节课讲到哪、课程目标是什么。

所以你的每一节课都长这样：

```
[ 30 分钟开场 ]
  - 我是谁，你是谁
  - 我们这门课的目标
  - 之前 5 节课讲过的关键点
  - 今天可以用到哪些参考资料

[ 30 分钟新内容 ]
  - 今天真正要讲的东西
```

每天前 30 分钟讲的话**几乎一字不差**，但你不得不每天讲一遍——因为他不记得。

LLM 就是这个学生。模型本身没有跨调用的记忆，每次 API 请求都必须把"开场那一大段"重新塞进去 prompt。这一大段在工程上很贵（见[深入 04 · 为什么简单"你好"也消耗数万 Token](04-为什么简单你好也消耗数万token.md)）。

> **Prompt Caching 做的事**：把"开场那段重复讲的话"录下来。下次上课，只要开场一字不差，就**直接从录音的末尾接着讲新内容**，省掉前 30 分钟的"现场讲述成本"。

这就是这一节要讲清楚的全部直觉。剩下的内容都是在解释：**录音具体录了什么 / 怎么判定一字不差 / 录音保存多久 / 录音失效了怎么办**。

## 1. 一个具体场景：RAG 客服的账单为什么差 8 倍

你做了一个客服助手，工作模式是：

```
[ 系统提示 5k token ：你是 X 公司的客服 ... ]
[ 知识库节选 30k token ：产品手册前 60 页 ... ]
[ 用户问题 200 token ：我的订单为什么还没发货？ ]
```

每个用户的问题不同，但前面 35k token 是**完全一样**的（同一个知识库节选）。

不开 caching 时，每次请求服务都要：
- 把全部 35,200 token 喂进模型，从头算一遍——这一步叫 **prefill**
- 然后才开始生成回答

开 caching 时：
- 第 1 次请求：照常算完整 35,200 token，但顺手把前 35,000 token 的"中间结果"存下来
- 第 2 次开始：前 35,000 token 跳过计算，**直接读取存下来的中间结果**，只对最后 200 token 做 prefill
- 计算量从 35,200 → 200，**大约 175 倍的省**

按 Anthropic 当前定价折算成钱：
- 命中缓存的 token：约 **0.1×** 常规价（省 90%）
- 写入缓存的 token：约 **1.25×** 常规价（多付 25%，因为要存一份）
- 没缓存的 token：常规价

如果同一份知识库一天被命中 1000 次，第 1 次贵了 25%，后 999 次每次省了 90%——账单大约下降 **80%–88%**。

这就是为什么这个东西值得一整节去讲。

## 2. 录音里录的到底是什么：不是回答，是"中间状态"

很多人第一反应是："缓存的是问题对应的答案吧？"——**错了**。

如果缓存的是答案，那就解释不了：
- 为什么 `temperature > 0` 时同一个 prompt 命中缓存，回答还会不一样？
- 为什么换一个用户问题，前面的缓存还能复用？

真相：缓存的是**模型在读完前面那段话之后，脑子里形成的中间状态**。这个中间状态叫 **KV cache**（K = Key 张量，V = Value 张量，是 Transformer 注意力机制里的两个中间产物）。

你可以把它理解成：

> 模型读了 35k token 的产品手册之后，在"工作记忆"里整理出了一份索引。Prompt Caching 把这份索引存下来。下一次同样的 35k token 进来，**不用重读手册、重建索引**，直接把索引调出来用，然后基于这份索引去回答今天的新问题。

**关键事实**（不需要记忆，知道有这回事就行）：
- 这份"索引"的体积**比原文大**——70B 模型，1k token 的输入，大约对应 **几十 MB** 的 KV cache，存在 GPU 显存或主机内存里
- 所以缓存**占显存**，会和"能服务多少并发"互相挤占（详见 §6.3）
- 索引的内容**只对当前模型有效**：换模型版本、换量化方式，索引全部作废

## 3. 录音怎么对上：Token 级前缀匹配

这是最多人踩坑的地方。**"一字不差" 不是指字符串相等，是指 token 序列的前缀相等**。

### 3.1 为什么是 token，不是字符

模型不直接处理字符，先把字符切成 token。「Hello world」可能是 2 个 token，「Hello, world!」可能是 4 个，多一个空格甚至可能影响切法。

判定缓存是否命中，**比的是切完之后的 token 序列**。

### 3.2 为什么是"前缀"，不是任意位置

模型读 prompt 是**严格从左到右**的。第 K 个 token 的中间状态，依赖前面 1~K-1 个 token——前面的任何一个 token 改了，第 K 个 token 的中间状态就要重算。

所以：
- ✅ 「[A][B][C][D]」的缓存可以复用 → 给「[A][B][C][E]」用前 3 个 token 的部分（前缀 ABC 一样）
- ❌ 不能复用给「[X][B][C][D]」（前缀第一个就不一样了，后面所有 token 的中间状态全部要重算）
- ❌ 不能复用给「[A][X][C][D]」（第二个 token 一改，后面 C/D 的中间状态都得重算）

### 3.3 工程上的一句话规则

> **越靠前的内容应该越稳定，会变化的内容尽量往后放。**

把这句话翻译成具体不该做的事：
- ❌ 系统提示开头放当前时间戳 → 每次请求时间戳都变，整个缓存全废
- ❌ 系统提示开头放用户 ID → 每个用户独享一份缓存，复用率从「按业务级」降到「按用户级」（不一定是 bug，但要算清楚）
- ❌ 系统提示开头加版本标签做 A/B 测试 → 切版本时缓存全部失效

应该这样：
- ✅ 固定的系统提示放最前
- ✅ 长期不变的参考文档接在后面
- ✅ 会变的当前对话、当前问题放最后

### 3.4 一个反直觉的细节：尾部多一个空格也会失效

`"hello world"` 和 `"hello world "`（多一个空格）很多模型分词结果不一样。看起来一样，token 序列不一样，缓存就 miss。

实战做法：**别假设"看起来一样"就 ok，必须用真实流量回放对一遍**。

## 4. 三家厂商怎么做：从用户视角对比

**一句话记住三家区别**：Anthropic——你告诉它"这段帮我记住"；OpenAI——它自己猜要不要记；Gemini——你把要记的东西上传成一个文件。下面逐个拆开看。

机制原理是相通的，但用法、定价、控制粒度差别很大。

### 4.1 Anthropic：客户端显式打标记

你在请求里告诉它"这一段我希望缓存"：

```json
{
  "model": "claude-opus-4-7",
  "system": [
    {
      "type": "text",
      "text": "你是一个 SRE 助手...（长系统提示）",
      "cache_control": {"type": "ephemeral"}
    }
  ],
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "参考文档：...（几万 token 的文档）",
          "cache_control": {"type": "ephemeral"}
        },
        {
          "type": "text",
          "text": "问题：这份文档里的 X 是什么意思？"
        }
      ]
    }
  ]
}
```

两个 `cache_control` 标记了两个**断点**。服务会：

1. 算到第一个断点 → 把这一段的"中间状态"存下来
2. 算到第二个断点 → 把这一段的"中间状态"也存下来（追加，不覆盖前一段）
3. 对最后的"问题：..."正常做 prefill
4. 生成回答

下次请求**前两段一字不差**，直接命中两段缓存，从最后的"问题"开始算。

**特点**：可控、便宜（命中省 90%），但写入要多付 25%。**重用 ≥ 2 次就回本，越多越值**。

### 4.2 OpenAI：自动开，自动判

不用打标记。当 prompt ≥ 1024 token 时自动尝试缓存，按 128 token 粒度做前缀匹配，组织级共享。

**特点**：省事，但**你不知道这次到底命中没**——只能从计费回包里反推。命中省 50%，写入 0 溢价。

### 4.3 Google Gemini：缓存当成一个资源

更像"上传一份文件得到 ID"：

- 客户端先调一次 API 把要缓存的内容创建成 cached content，得到一个 ID
- 后续请求里引用这个 ID
- 按"缓存大小 × 存储时长"付费（不是按命中次数）

**特点**：适合"创建一次、查询很多次"的场景（比如把一本大书做成 cache，让用户对它问问题问一整周）。

### 4.4 对照表

| 维度 | Anthropic | OpenAI | Google Gemini |
|---|---|---|---|
| 触发方式 | 显式 `cache_control` | 自动（≥1024 token） | 显式创建 cached content |
| 最小缓存粒度 | 1024 token（Opus/Sonnet）/ 2048（Haiku） | 1024 token | 取决于模型，通常较大 |
| 写入成本 | ~1.25× 常规价 | 无溢价 | 按缓存大小 × 时长 |
| 读取成本 | ~0.1× 常规价（折扣 90%） | ~0.5× 常规价（折扣 50%） | ~0.25× + 存储费 |
| TTL | 5 分钟（命中续期；可延到 1 小时 beta） | 5–10 分钟（off-peak 可达 1 小时） | 用户自定 |
| 共享范围 | 同一 API key | 组织级 | 同一项目 |
| 最大断点数 | 4 | 不适用（自动） | 不适用 |

## 5. 录音什么时候作废：失效的五种形态

这是 SRE 最该关心的部分。**失效本身不会触发任何报警**——请求依然成功、依然返回正确答案，只是延迟变长、账单变贵。所以必须主动监控。

### 5.1 TTL 过期（时间到了）

每段缓存有保质期，通常 5 分钟。Anthropic 是**命中续期**——只要持续有请求命中，TTL 一直被刷新；一旦冷下来，到点失效。

**踩坑**：低频调用场景（每 10 分钟一次）**永远命中不了**，开了 caching 等于白付写入溢价。

**对策**：要么不开缓存，要么用心跳请求维持，但要算这个心跳值不值（心跳本身花钱）。

### 5.2 路由变了（实例换了）

推理服务是分布式的，缓存通常**只存在于具体一台 GPU 实例的内存里**。如果下次请求被路由到另一台实例，prompt 完全一样也会 miss。

云厂商通常会尽量做"sticky 路由"（同 session 黏到同实例），但**没有 SLA 承诺**。流量突增、实例扩缩容时命中率会下降。

**自建推理服务时**：必须自己设计路由策略（按 session ID、按 prompt 前缀 hash），否则缓存机制名存实亡。

### 5.3 前缀变了（前面动了）

§3.3 已经讲过：前面任何一个 token 变化都会让后面全部失效。生产里最常见的几个犯罪现场：

- 系统提示里嵌了当前时间或当天日期
- 系统提示里嵌了用户 ID / 会话 ID
- 系统提示开头加了版本号 / A/B 标签
- 中间插了一段动态生成的"今日提示"

### 5.4 实例重启 / OOM

部署新版、crash 恢复、扩容新起的实例 → 缓存为空，需要"预热期"才能恢复命中率。**部署窗口期成本和延迟都会上扬**——发版前一定要预期到。

### 5.5 模型版本切换

缓存是**模型绑定**的。换模型版本（甚至换量化方式）后，旧缓存全部作废。**灰度发布期间命中率会两端都下降**：旧版流量减少导致旧缓存冷却失效，新版还没攒起新缓存。

## 6. 对 SRE 的工程含义

### 6.1 必须监控的指标

| 指标 | 说明 | 警戒线 |
|---|---|---|
| Cache hit rate | 总命中率 | 低于业务预期 20% 就该报警 |
| Cache hit prefix length | 平均命中多少 token | 突然变短 → 前缀设计被某个改动破坏了 |
| Cache write rate | 每秒写入次数 | 突增 → 前缀在变（有人加了动态字段） |
| Per-session/per-channel hit rate | 单会话 / 单上游通道 | 某些维度命中率异常低，针对性排查 |
| Cost savings | 按 token 折算的节省 | 业务层追踪 |

### 6.2 路由设计（自建推理服务时）

- **按 session 路由**：同会话黏到同实例，命中最优
- **按 prompt 前缀 hash 路由**：共享前缀的请求黏到同实例
- 代价：负载均衡变差，可能产生热点。需要在"命中率"和"均衡性"之间权衡。

### 6.3 容量规划：缓存和并发是此消彼长的

缓存**占显存**。一个 70B 模型、128k context 的完整 KV cache 大约 **40 GB**。

> **给缓存留多少显存 ↔ 能服务多少并发——此消彼长。**

留得多：命中率上升、单次请求便宜、首包延迟低；但能跑的并发数下降。这是 AI 时代容量规划的新维度。

### 6.4 失效的静默性

再强调一次：缓存失效**没有报警**。
- 请求还在正常成功
- 延迟变长（但不到 timeout 阈值）
- 成本变高（但财务数据有滞后）

**只能靠主动监控 hit rate**，不能等出账单时才发现。

### 6.5 多上游网关位的特殊风险

> **一句话**：网关挂了三家上游，缓存策略不能"一刀切"。同一段 prompt 走 Anthropic 和走 OpenAI，缓存账单可能差 50%——网关不知道是哪类工作负载就做自动路由，等于在烧钱。

如果你做的是**通用 LLM 网关**（一个 endpoint 后面挂着 OpenAI / Anthropic / Gemini / 自建模型若干家），三家差异就从"接口细节"升级成"生产风险"。

**风险一：写入溢价不对称**

把同一个用户从 OpenAI 通道切到 Anthropic 通道，看似只换了 model 名字，账单完全不同：

- OpenAI 通道：写入 0 溢价，命中省 50%
- Anthropic 通道：写入溢价 25%，命中省 90%

对**高重用场景**（多轮对话、RAG 反复命中），切到 Anthropic 反而便宜；对**低重用场景**（一次性长文档分析），切到 Anthropic **直接涨价 25%**。网关如果不区分这两类工作负载就做"自动路由"，账单会无缘无故上下浮动。

**风险二：归一化是个伪命题**

"在网关层抹平 caching 差异"听起来漂亮，做不到：

- Anthropic 要客户端显式打 `cache_control`，OpenAI 不需要 → 网关无法用同一种入参表达"我希望缓存这一段"
- OpenAI 命中是 prefix 自动判定的，网关读不到"这次到底命中了没"——只能从计费字段反推
- Gemini 的 cached content 是**显式资源**（先 create 拿 id，再引用），跟前两家的"内联标记"模型根本不同

**结论**：网关层**只能做透传 + 观测，不要做归一化**。

**风险三：计费回放与对账**

每家上游回包里"缓存命中量"的字段名都不同：
- Anthropic：`usage.cache_read_input_tokens / cache_creation_input_tokens`
- OpenAI：`usage.prompt_tokens_details.cached_tokens`
- Gemini：不在响应里，要去查 cached content 的 metadata

网关如果用一套统一 `usage` 结构落 DB，**默认会丢一两个字段**，月底对账对不上。

**网关侧最小自检**：

- [ ] 计费日志同时记录 `prompt_tokens / cache_read_tokens / cache_write_tokens / completion_tokens` 四列。任一上游缺字段就填 0，**字段必须存在**
- [ ] 不同 channel 的 hit rate 分开看，混在一起没意义
- [ ] 跨 channel 路由策略上线前，**先用真实工作负载回放算两边账单**，再决定是否切流
- [ ] 跟客户沟通时讲清"换通道 = 换缓存计费模型"，别让调用方以为"模型名一样就一样"

## 7. 实战：哪些场景最受益

**高受益场景**——前缀长、重复用、变化只在尾巴：
- **RAG**：固定文档放前面，变化查询放后面
- **长系统提示的对话产品**：Few-shot 示例、人设定义放前面
- **代码助手**：仓库/文件上下文放前面
- **多轮对话**：前几轮历史可以被缓存，每次断点往后推进

**低收益甚至负收益**：
- 一次性短 prompt
- 每次前面都在变的场景（嵌了时间、随机值、用户 ID）
- 调用频率比 TTL 还低（每 10 分钟一次以下）→ 写入溢价交了，命中率约等于 0

---

## 8. 常见陷阱 Checklist

- [ ] 不要在 prompt 最前面放时间戳 / UUID / 随机值
- [ ] 不要每次改版本号放在系统提示开头
- [ ] 用户 ID / session ID 没必要就别放前面（放前面 = 缓存按用户隔离）
- [ ] **要监控实际命中率，不要假设设了 `cache_control` 就一定命中**
- [ ] 模型灰度发布时预期到缓存失效的成本冲击
- [ ] 自建服务时设计路由亲和，否则缓存机制名存实亡
- [ ] 不要以为缓存的是"答案"——`temperature > 0` 时同 prompt 答案也不同，缓存的是中间状态（KV cache）
- [ ] 写入比读取贵——冷门 prompt 不要标记缓存
