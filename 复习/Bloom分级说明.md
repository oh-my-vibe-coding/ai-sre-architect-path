# Bloom 分级说明 · 本书练习的认知层级

> [← 复习系统](README.md)  ·  [← 返回目录](../README.md)

> [!NOTE]
> Bloom's Taxonomy（1956，2001 修订版）是教育学最被广泛使用的**认知层级**框架。
>
> **为什么标出来**：让你看到自己在哪一层——避免"以为懂了"但一直停在"理解"。

---

## 1. Bloom 六层

```
        创造 (Create)       ↑ 难
        评估 (Evaluate)      │
        分析 (Analyze)       │
        应用 (Apply)         │
        理解 (Understand)    │
        记忆 (Remember)      ↓ 易
```

| 层级 | 动作词 | 本书例子 |
|---|---|---|
| **记忆 Remember** | 列出、命名、回忆 | 列出 Lethal Trifecta 三个组成 |
| **理解 Understand** | 解释、用自己的话、举例 | 用自己的话解释量化为什么可能失败 |
| **应用 Apply** | 使用、计算、演示 | 对你的场景做一次 trifecta 分析 |
| **分析 Analyze** | 对比、拆解、找关系 | 对比 AWQ 和 GPTQ 的思路差异 |
| **评估 Evaluate** | 判断、审查、挑战 | 审查某架构方案的完备性 |
| **创造 Create** | 设计、提出、整合 | 为新业务设计 AI 自治策略 |

## 3. 每个 Unit 的 Bloom 目标（推荐）

| Unit | 核心产出 | 目标层级 |
|---|---|---|
| **Unit 0 · API 上手** | 可跑的 CLI Agent | 应用 |
| **Unit 1 · Agent 自治** | 架构约束设计文档 + 红队自检 | **综合 + 评估** |
| **Unit 2 · Trace-Eval** | 统一观测设计图 + PoC | **综合** |
| **Unit 3 · 推理 SLO** | SLO + 容量 + 降级检测方案 | **综合** |
| **Unit 4 · 复合可靠性** | 评审文档 + 失效放大分析 | **评估 + 综合** |
| **Unit 5 · 数值调试** | Runbook | 应用 + 分析 |

**关键**：Unit 1 之后所有主产出都在**综合**或以上。如果你做出来的东西只停在"应用"层（照着模板填），说明没达到 mastery（见 [Mastery Gates](#5-mastery-gates)）。

## 3.5 非 Unit 章节的建议复习层级

Unit 是"必须做到"的层级（mastery gate），非 Unit 章节是"读到就有用"的资源——但仍可以用 Bloom 给出**建议复习深度**，避免对所有章节用一样的劲。

| 章节系列 | 建议层级 | 触达方式 | 例子 |
|---|---|---|---|
| **理念 01-03** | 理解 | 通读一次 + 能用自己的话复述 | 给同事讲"AI 时代 SRE 的角色迁移" |
| **共同语言 01-05** | 记忆 + 理解 | SR 卡刷词汇 + 能在跨团队对话中**听懂不冷场** | 别人说 "RLHF / Recipe / Eval set"，能跟上 |
| **深入 01-04（推理性能）** | 应用 | 用公式算一次自己的系统 | KV cache、TTFT、prefix cache 命中率算到具体数字 |
| **深入 05（容量规划）** | 应用 + 分析 | 给真实系统做完整 worked example | 沿着深入 05 §7 的步骤算 SRE 助手 |
| **深入 06（Eval）** | 应用 + 分析 | 给当前系统设计 L1/L2 + 跑起来 | RAG 系统的 eval pipeline |
| **深入 07（红队）** | **评估** | 实际对系统做一次红队 | 至少跑出 3 个攻击链 |
| **深入 08-12** | 理解 + 选择性应用 | 通读 + 命中场景时回查 | "现在要选型"时回到深入 12 |
| **科学 01-04** | 理解 | 通读 + 复述机制 | 能解释 Lost in the Middle 为什么发生 |
| **知识 04-09**（章节主体） | 应用 + 分析 | 配套 Unit 走完 | 每章对应一个 Unit 的 mastery gate |

> **怎么用这张表**：复习时先看自己**实际触达**到哪一层，再看建议层级——只有当**实际 < 建议**时才需要补。深入 07 是唯一一章建议达到"评估"层的非 Unit 内容，因为红队本身就是评估行为。

## 4. 如何自我诊断"卡在某一层"

### 卡在"记忆 / 理解"的症状
- 能背概念但**没法用自己的话讲给同事**
- 能复述书里的例子但**举不出自己的例子**
- **应对**：做 Active Recall 题库里的"用自己的话解释"

### 卡在"应用"的症状
- 照着模板做，但**改一个条件就不会了**
- 能做**单一任务**但**跨领域组合时崩**
- **应对**：做"跨章综合题"

### 卡在"分析"的症状
- 能对比两个方案，但**给不出选哪个的判据**
- **应对**：强迫自己写 decision document（Why A over B）

### 跃升到"评估 / 创造"
- **没法一个人凭空跃升**
- 需要**真实场景 + 反馈**——这就是 [练习](../练习/) 部分的 Unit 产出 + 红队的设计目的

## 5. Mastery Gates

每个 Unit 的产出应达到**对应 Bloom 层级**才视为 mastered。

### Unit 1 · Mastery Gate 示例
产出：架构约束设计文档 + 红队自检。**达到"综合 + 评估"层** 需要：

- [ ] Trifecta 分析**覆盖所有已知输入面**（不漏任何一条工具、一条外泄通道）
- [ ] 3 套方案各自有**具体的劣势 / 代价**分析（不是"看起来都好"）
- [ ] 分级表**Forbidden 不为空**（有些操作必须禁止，不能全自治）
- [ ] 红队至少发现 **3 个漏洞**，其中 **1 个 AI 没想到**
- [ ] 至少做过 **2 轮修改**（初稿 → AI 挑错改 → 自己或同事再挑改）

**达不到 gate，不进 Unit 2**。硬卡。原因：Unit 2 的复杂度建立在 Unit 1 的判断力上。

## 6. "分层学习"的实操建议

### 刚学一个新概念（记忆 / 理解）
- 读章节
- 过 SR 卡
- 给同事讲一遍 / 给 AI 讲一遍

### 开始用这个概念（应用）
- 做 Unit 里的 Week 任务
- 在真实工作中找一个场景应用

### 想真正掌握（分析 / 评估）
- 做 Active Recall 题库
- 对比多个方案
- 参与代码 / 架构评审

### 想创造（创造）
- 带着这个概念做一个端到端项目
- 写博客 / 开源工具
- 在新场景里迁移

---

## 7. 给 SRE 的一句话总结

> [!IMPORTANT]
> 很多自学者的失败不是"学得不够多"，是"**一直停在理解层，错把理解当掌握**"。
>
> Bloom 分级让你每周都能问自己：**"我这周停在哪一层？该往上推一层吗？"**
>
> 4-5 个月学习周期里，真正的差距不是时间，是**是否把层级推到了评估 / 创造**。
