# SRE 行为反模式 Top 10

> 全书各章在讲"系统会出什么故障"（深入 10 的事故模式库）。这份清单讲的是 **SRE 自己会犯什么错**——那些看起来合理、实际上会把你带进坑里的行为和决策模式。
>
> 每月自检一次：本月我踩了几个？

---

## 反模式 1 · 过早自建推理服务

**表现**：上来就要搭 vLLM + GPU 集群，觉得"用 API 不专业"。

**为什么是反模式**：
- 你不知道标准模型能做到什么，就不知道自建要对齐到什么基线
- 运维 GPU 集群的学习曲线比你想象的大一个数量级
- 在你把 vLLM 调稳之前，业务需求已经变了三轮

**正确做法**：引章里写得很清楚——**先把 API 用熟，再按需下沉**。自建的理由应该是成本、延迟、合规，不是"别人都自建"。

**来源**：[引章 · §你应该选什么服务](../01-引章-大模型速览.md#你应该选什么服务)

---

## 反模式 2 · 用在线推理的思维跑批处理

**表现**：把离线数据标注/评估任务按在线 API 的标准做——关注延迟、用 on-demand GPU、不利用 batch API 的半价。

**为什么是反模式**：
- 批处理场景吞吐 > 延迟，Spot 实例 > On-demand，Batch API > 在线 API
- 一个组织里离线推理的 token 消耗通常是在线的 3-10 倍，但成本思维还是在线的
- 在线推理的 SLO 套到离线场景 = 烧钱最快的方式

**正确做法**：评估库存里有"批处理占比"这个指标。离线任务走 Batch API 或 Spot 实例，主动设计前缀复用顺序。

**来源**：[深入 13 · 离线批量推理](../深入/13-离线批量推理.md)、[深入 18 · 成本工程](../深入/18-LLM成本工程.md)

## 反模式 3 · 把 eval 当一次性工具

**表现**：上线前跑一次 eval，通过就上线，然后半年不碰。

**为什么是反模式**：
- 模型悄悄升级（厂商不会通知你）、prompt 累积改动、输入分布漂移——六个月后你的 eval 结果已经和线上表现没关系了
- 这是 Anthropic 2026 三连事故的核心教训——"你的 eval 会骗你"

**正确做法**：Eval 是持续运行的生产监控，不是上线前的 checkpoint。给它定 SLO（覆盖率、延迟、Judge κ、pipeline uptime）。Eval 挂了 = 质量盲眼。

**来源**：[第 7 章 · 质量可观测性](../知识/07-质量可观测性与DataFlywheel.md)、[深入 06 · Eval Pipeline](../深入/06-Eval-Pipeline设计.md)

## 反模式 4 · 把 prompt 安全寄托在应用层

**表现**：用 LLM 分类器做输入安全检查、在 prompt 里写"不要回答有害请求"、指望 Agent "自律"。

**为什么是反模式**：
- 分类器本身也是 LLM，会被更聪明的 prompt injection 绕过
- Prompt 里的安全规则对 Agent 来说就是另一段文本，没有强制力
- 物理边界才是不可绕过的——你信任 Agent 的自律，就像你信任用户的善意

**正确做法**：安全在基础设施层强制——沙箱、egress 白名单、capability-scoped 凭证。砍掉致命三角的至少一条腿。

**来源**：[第 6 章 · 致命三角](../知识/06-AI自治与上下文架构约束.md)、[深入 07 · 红队实战](../深入/07-Agent-Prompt-Injection红队实战.md)

## 反模式 5 · 不给上下文做预算

**表现**：Agent 链无限加 tool call，追求"做得更多"但没算过步数成本。Context 膨胀了就换更大的 context window。

**为什么是反模式**：
- 每多一个 tool call，就多一次 latency + token 成本 + 错误概率乘积
- Context window 名义上 1M 但实际有效 ~64K——加了也没用（Lost in the Middle）
- 步数预算和 P99 latency budget 是同一类工程约束——不列出来它就不存在

**正确做法**：给每条业务链路定 **step budget**（最多 N 步）。每多一步要有边际收益交代。超预算的 step 裁剪或合并。Context 超过"有效拐点"时触发 compaction 而非加长。

**来源**：[第 4 章 · Step 预算](../知识/04-系统架构与复合AI可靠性数学.md)、[科学 02 · Lost in the Middle](../科学/02-Lost-in-the-Middle.md)

## 反模式 6 · 只看整体平均，不看分桶

**表现**：看"全局幻觉率 5%"觉得还不错，但没发现某类任务错误率已经 30%。

**为什么是反模式**：
- LLM 的失败高度集中在特定类别上——量化后的数字类任务、长 context 的中段、特定语言的 tokenization 劣化
- 平均指标掩盖了尾部——而事故就发生在尾部
- 量化静默降级的典型症状就是"整体 eval 只降 1-2%，某类任务崩了"

**正确做法**：质量指标按任务类型、context 长度、语言分桶。找到"掉得最多的那一类"——那才是真正的故障面。

**来源**：[科学 03 · Quantization](../科学/03-Quantization为什么有时坏.md)、[第 5 章 · 静默降级](../知识/05-AI推理服务的可靠性工程.md)

## 反模式 7 · 模型版本管理靠"latest" tag

**表现**：推理服务指向 `my-model:latest`；厂商升级后行为悄悄变了，排查只能翻 changelog。

**为什么是反模式**：
- "Latest" 不是版本——是"我放弃版本控制"的声明
- 厂商悄悄升级模型、你悄悄改了一行 prompt、embedding 模型被偷换——任何一个都可能导致静默降级，而且你不知道什么时候开始的
- 回滚时发现"上一版"已经不存在了

**正确做法**：所有模型（LLM、embedding、reranker、judge）全部 pin 具体版本或 sha256。灰度用版本号而非百分比。永远不要自动删除生产模型，归档而非删除。

**来源**：[深入 15 · Model Registry](../深入/15-模型注册表与上线流程.md)、[深入 16 · Embedding 服务](../深入/16-Embedding-服务作为独立运维对象.md)

## 反模式 8 · 把 RAG 当成一个整体盯

**表现**：监控"RAG 端到端响应时间"但不知道慢在 embedding 还是 vector DB 还是 LLM。

**为什么是反模式**：
- RAG 是四个独立运维对象：Embedding 服务、Vector DB、Reranker、LLM
- 每个有自己的 SLO、容量模型、升级影响面
- 把四个对象当一个东西盯 = 不知道谁的 SLO 先崩
- RAG 失败的根因 60%+ 在 embedding/vector 层，但注意力 90% 在 LLM 层

**正确做法**：给四个组件各自定 SLO，trace 串起来，分 span 看延迟和错误。Embedding 的 retrieval 质量指标（Hit@k、MRR）和 LLM 的回答质量指标是两套独立的监控。

**来源**：[深入 16 · Embedding 服务](../深入/16-Embedding-服务作为独立运维对象.md)

## 反模式 9 · 缓存命中率不监控，等账单才发现

**表现**：设了 `cache_control` 就以为缓存生效了。缓存因为 prefix 里加了时间戳而全部失效，但请求正常返回、延迟正常——直到月底账单出来。

**为什么是反模式**：
- 缓存失效是静默的——没有 error、没有 timeout、没有告警
- 唯一能发现的窗口是 cache hit rate 指标
- 缓存失效的成本冲击往往是 2-10 倍，但财务数据有 N 天滞后

**正确做法**：Cache hit rate 是 P1 监控指标，和 p99 latency 同级。掉 20% 就该告警。模型版本灰度、prompt 结构调整后，主动核查 hit rate。

**来源**：[深入 02 · Prompt Caching](../深入/02-Prompt-Caching原理.md)、[深入 18 · 成本工程](../深入/18-LLM成本工程.md)

## 反模式 10 · 把成本推给财务，不把它当架构指标

**表现**：唯一看成本的地方是月末报销单。日常 Dashboard 有延迟、有错误率、有 QPS，但没有成本。

**为什么是反模式**：
- Token 成本是工程行为的函数——系统提示多长、Agent 走几步、缓存命中多少
- 成本异常往往是架构问题的第一个信号（Agent 死循环、context 膨胀、缓存失效、模型误升级到旗舰版）
- 等月末才发现成本暴涨 = 你已经瞎了三周

**正确做法**：成本是 Dashboard 的一等指标，和延迟/错误率同级。按日、按模型、按租户分桶。日成本突增 30% 触发告警。成本不是财务指标——是你的**架构健康度指标**。

**来源**：[深入 18 · LLM 成本工程](../深入/18-LLM成本工程.md)
