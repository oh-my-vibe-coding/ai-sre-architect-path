# Unit 5 · 数值 / 编译器级 LLM serving 调试（2 周）

> [← 返回目录](../../README.md)  ·  对应知识章节：[第 9 章](../../知识/09-工程底座.md)

## 目标
能在 **bf16/fp32、kernel launch、XLA IR** 层做故障排查。这是最后一个单元，也是最窄最深的一块——**别人不会的时候，它是你的护城河**。

## 产出
数值级故障**排查 runbook**（模板化，供后续 on-call 使用），包含：
- 常见数值问题的识别信号
- 分层排查流程（从应用日志 → kernel → 编译器）
- 最小复现实验的设计方法

**Bloom 目标层级**：**应用 + 分析（Apply + Analyze）** — 能按方法论排查，能拆解复杂故障。

## Mastery Gate

达到以下标准才算 Unit 5 mastered（也是全书最后一个 gate）：

- [ ] Runbook 能让**另一个没参与的同事**独立排查至少 2 类数值故障
- [ ] 至少做过**一次真实或模拟**的数值级 debug（不是纸上谈兵）
- [ ] 分层流程**每一层都有具体命令 / 工具**（`nvidia-smi` / kernel profile / IR dump）
- [ ] 能解释**为什么 Anthropic "三次事故"复盘里某个 bug 属于数值层面**
- [ ] 最小复现方法论**可应用到新故障类型**（不是一次性脚本）

**未达标的表现**：Runbook 抽象 / 没做过真实排查 / 分层只有名字没有命令 / 不能迁移到新故障。

## 一手材料

- NVIDIA · CUDA determinism guide
- Anthropic · Postmortem of three recent issues（第三次重读，这次从数值角度读）

## 不同硬件条件的学习路线

> [!TIP]
> **本 Unit 不只服务"有 GPU 的读者"**。A-C 三档路线都能完成 Mastery Gate 的核心目标（理解数值层、能写 runbook、能在 on-call 里认出这类问题），只是动手深度不同。D 档适合管理者或暂时没有实验条件的读者，但不完全满足"动手排查"这一项。
>
> **鼓励原则**：**有替代路径，但不能只看不想**。每档都要产出实验记录或 runbook。

### 路线 A · 有 GPU（本地或公司资源）
- 跑 7B/8B 模型做完整实验 A/B/C
- 重点：观察 **bf16 vs fp32、kernel 非确定性、长 context 精度累积** 的端到端效应
- 产出：Runbook **含实测数据**

### 路线 B · 有云资源（Colab / Modal / 短期云 GPU）
- 用 **Colab / Kaggle / 云 GPU** 跑简化版实验；T4 更适合 7B 量化或小模型，bf16/full precision 建议用 L4/A10/H100 级别资源
- 或 Modal / RunPod 按量买 1-2 小时 H100
- 产出：Runbook **含小规模实测**

### 路线 C · 无 GPU（笔记本 CPU / Apple Silicon）
- 用 **小模型**（Qwen3-1.5B / Phi-3-mini / Ministral-3B）在 MacBook 跑 bf16 推理
- Apple Silicon 可用 **MLX** 框架，推理比 CPU 快 10 倍
- **Toy softmax** + **bf16/fp32 小矩阵** + **seed determinism** 实验
- 产出：Runbook **明确标注"未在 frontier 规模实测"**；CPU 层概念演示代码

### 路线 D · 只读 / 管理者路线
- 不跑任何代码
- 专心读 Anthropic 事故复盘 + 本书 [科学 03](../../科学/03-Quantization为什么有时坏.md) / [深入 10 · Pattern 1/12](../../深入/10-AI系统事故模式库.md)
- 产出：**事故模式映射**（把 3 次 Anthropic 事故对应到本书 Pattern）+ **Runbook 框架**（不含实测命令输出）
- 适合：技术管理者、架构评审者、暂时不碰代码的 SRE

### 路线推荐

| 你的情况 | 推荐路线 |
|---|---|
| 负责 LLM 服务运维 | A 或 B（**必做实测**）|
| 通用 SRE，偶尔碰 LLM | B 或 C |
| 学习 + 观察 | C |
| 技术管理 / 架构评审角色 | D（但鼓励**挑 1 个 toy 实验**动手） |

**无论哪档，最终目标不变**：能在 on-call 里认出这类问题 + 能产出可用的 Runbook。若走 D 档，请在后续补一次 toy 实验或云端实测，再算真正完成 Mastery Gate。

> **D 档补做触发点**（任一发生就动手）：
> - (a) 公司有 GPU 资源后
> - (b) 下次 on-call 遇到任何质量类工单时（哪怕只是 5 分钟在终端跑一次 toy softmax bf16 对比）
> - (c) 半年自检（[附录 A](../../附录/A-每月自检表.md)）时若仍未补，把"Unit 5 未结"写进备注栏
> 补做后更新 [Capstone § 11](../Capstone-AI生产架构评审包.md)（数值级故障 Runbook），把"未实测"声明删掉。
