# Unit 1 · Week 1 · 致命三角初识

> [← Unit 1 总览](总览.md)  ·  [← 返回目录](../../README.md)

## 本周目标
掌握 **Lethal Trifecta** 威胁模型，并用它分析你**自己当前系统**里的一个真实场景。

> **前置概念**：致命三角的权威定义见 [第 6 章 · AI 自治与上下文架构约束 · 致命三角](../../知识/06-AI自治与上下文架构约束.md#致命三角lethal-trifecta)。本周练习直接使用其结论，不再重复定义。红队视角下的"找完整攻击链"方法见 [深入 07](../../深入/07-Agent-Prompt-Injection红队实战.md)。

## 阅读 · B3 · 45 分钟（无 AI）
**Simon Willison · 《The lethal trifecta for AI agents: private data + untrusted content + external communication》**

固定 URL：https://simonwillison.net/2025/Jun/16/the-lethal-trifecta/
若 404，搜 "lethal trifecta site:simonwillison.net"。

阅读要求：
- 不开 AI 辅助、不用翻译插件
- 手边放笔记，遇到"让你停顿超过 3 秒的句子"就记下来
- 45 分钟到就停，没读完也停——**练的是连续专注，不是读完**

读完应该能复述：
- 为什么是这三件东西共存才有风险，两件单独各自没事？
- "external communication" 为什么是最容易被忽略的一条？

## 产出 · B2 · 45 分钟（不用 AI 写）
对你当前工作中**一个"让 AI 操作线上"的真实场景**，自己写清楚三件事：

### 1. 不受信输入是什么？
候选来源清单，挑出你场景里至少 2 种：
- 终端用户的 prompt
- 从数据库 / 日志 / 工单系统读进 prompt 的文本
- 外部 API 响应（搜索结果、爬取内容）
- 其他 Agent / Tool 的输出

**标准**：只要存在"内容来自模型信任边界以外"的可能，就算不受信。

### 2. 工具访问范围多大？
列出 Agent 能执行的**所有**操作：
- 读取哪些数据源？
- 修改哪些系统？
- 调用哪些外部 API？
- 执行 shell / 代码的权限？

### 3. 外泄路径在哪？
Agent 如果被恶意诱导，数据能送到哪里去？
- 写入日志 → 谁能看到这些日志？
- 调用出站 API → 对方能不能解析到敏感信息？
- 构造带数据的 URL 让用户点击
- 通过 error message 回显

## 写完之后：让 AI 挑错
把三部分分析贴给 AI：

> "这是我对 X 系统 lethal trifecta 的分析。请挑漏洞：我有没有遗漏输入来源？工具列表是否完整？外泄路径有没有没想到的？不要重写，只指出问题。"

根据反馈**自己改**，不要让它直接重写。

## 预测 · B1 · 每日 5 分钟
本周每次**设计/讨论 AI 相关需求**时，先猜：
- 这个设计有没有构成致命三角？
- 如果有，哪条腿最容易被现有基础设施砍掉？

问完 AI 或同事后对比差距。

## 范例参考

> 以下是一个**填好的范例**（SRE 事故助手场景），让你看到"产出长什么样"。你做的会用你自己的场景，内容不同——但结构和深度参考这个。

### 1. 不受信输入

- **终端用户 prompt**：事故描述（"数据库慢了"、"502 了"），可能包含诱导性描述
- **从日志系统读进的文本**：`grep` 出的错误堆栈、kubectl describe pod 的输出——这些来自生产环境，可能被之前的错误日志污染
- **外部 runbook 内容**：如果 runbook 是 wiki 上共编的，任何人都可能在里面插入误导性指令

### 2. 工具访问范围

| 工具 | 权限 | 风险 |
|------|------|------|
| `kubectl get pods/logs/describe` | 只读，所有 namespace | 可能读到 secret 名、IP |
| `grep` 日志文件 | 只读，/var/log 下 | 可能读到 PII |
| `curl` 内部 API | 只读，内网 | 可能被诱导调不该调的 endpoint |
| `send_slack` | 写，发消息到频道 | **外泄通道** |

### 3. 外泄路径

- **Slack 消息**：Agent 如果被诱导把敏感日志内容发到公共频道 → 数据泄露
- **error message 回显**：Agent 调 API 失败时可能把 PII 打在错误信息里返回
- **带参数的 URL**：Agent 如果把"查到的工单详情"拼成 URL 让用户点 → 数据通过 referrer 泄露
- **没有外泄**：kubectl get 的结果只显示在终端，不落盘，不转发——这部分相对安全
