## 范例参考

> 关于 B2 产出的两个问题，这是一个**填好的范例**：

### 问题 1：Tool use 流程里，谁决定调不调用工具？谁决定用什么参数？谁决定执行？

- **决定调不调用**：模型。它看到用户说"磁盘用得满吗"，判断需要调用 `df -h` 来获取信息。这个判断来自 system prompt 里的工具说明——模型在 prompt 里"读"到了这个工具的能力描述。
- **决定用什么参数**：模型。它根据工具 schema（参数名、类型、描述）生成符合规范的 JSON 参数。比如 `run_shell` 要求 `cmd` 字段，模型就把 `"df -h"` 填进去。
- **决定执行**：你的代码。模型只返回一段 JSON（"我想调用 run_shell，参数是 df -h"）。你的代码收到后决定执不执行、怎么执行——这就是安全边界的位置。

### 问题 2：如果 AI 被诱导调用了错误参数，责任链条在哪一步断的？

**责任断在"执行层"**。模型的 tool_use 只是"建议"——它没有执行权。你的代码在收到 tool_use 后有三层可以截断：
1. **参数校验**：cmd 在白名单里吗？（只允许 uptime/df/ls/ps/free）
2. **沙箱**：即便 cmd 合法，它跑在受限环境里（容器、只读文件系统）
3. **审计**：每次调用都有日志，出事后可追溯

所以攻击者想让 Agent 执行 `rm -rf /`，需要同时绕过三层——模型不会提议（它在 system prompt 里被告知了工具的使用范围）、参数校验会拒绝（`rm` 不在白名单）、沙箱会阻止（没有写权限）。**安全不靠模型的自律，靠执行层的强制约束**。
