# Capstone · AI 操作生产环境架构评审包

> [← 返回目录](../README.md)  ·  [← 周循环总览](周循环总览.md)

> [!IMPORTANT]
> **这是本书学习的终点里程碑**。走完 Unit 0-5 之后，你要做的不是"再读一本书"，而是**把所学整合成一份可用于真实架构评审的文档**——放到你当前或下一个项目里，作为**可信背书**。

---

## 目标

把整本书的学习成果整合成**一份可对外评审的架构文档**：

- 对**你自己**：系统化检验 4.5 个月的学习是否扎根（终极 Mastery Gate）
- 对**你的团队**：作为真实架构评审的起点模板
- 对**你的事业**：作为简历 / 面试 / 晋升的硬产出

**不是抄书**。是把书学过的概念，**用你自己系统的真实信息**填进一份完整评审文档。

## 文档结构模板

### 1. 系统背景（1 页）

- 业务上这个系统做什么
- 当前规模（DAU / QPS / token 量）
- 团队配置（SRE / ML / 业务方）
- 评审的触发点（要上线？要扩容？要做安全审计？）

### 2. Lethal Trifecta 分析

> **参考**：[第 6 章](../知识/06-AI自治与上下文架构约束.md)、[深入 07](../深入/07-Agent-Prompt-Injection红队实战.md)、[Unit 1 · Week 1](Unit1-Agent自治与致命三角/Week1-致命三角初识.md)

- 不受信输入清单（**全部输入面**，一个不落）
- 工具访问清单
- 外泄通道清单
- 三角图
- 当前哪些腿已被砍 / 哪些还在

### 3. Agent 自治边界

> **参考**：[Unit 1 · Week 3](Unit1-Agent自治与致命三角/Week3-自治分级表.md)

- 自治分级表（Auto / Confirm / Forbidden）
- Forbidden 列表必须非空
- 每项的判据（可回滚性 / blast radius / 可审计）

### 4. Trace-Eval 一体化方案

> **参考**：[第 7 章](../知识/07-质量可观测性与DataFlywheel.md)、[深入 06](../深入/06-Eval-Pipeline设计.md)、[Unit 2](Unit2-TraceEval统一可观测性/总览.md)、[附录 E · 模板 1](../附录/E-模板库.md)

- 统一数据流图（trace ＝ eval dataset）
- Data Flywheel 流程图 + 每步 owner
- Eval 自身的 SLO

### 5. 三层 Eval + Judge 校准

> **参考**：[Unit 2 · Week 2-3](Unit2-TraceEval统一可观测性/Week2-三层Eval体系设计.md)、[附录 E · 模板 2](../附录/E-模板库.md)

- L1 Assertion 规则（≥5 条）
- L2 Judge 设计 + 校准方法（Cohen's κ 或 agreement rate）
- L3 A/B 分流策略
- Judge 偏见自检

### 6. 推理服务 SLO

> **参考**：[第 5 章](../知识/05-AI推理服务的可靠性工程.md)、[Unit 3 · Week 1](Unit3-推理SLO与静默降级/Week1-SLI定义.md)、[附录 E · 模板 3](../附录/E-模板库.md)

- 三类 SLI（延迟 / 容量 / 质量），每个有目标值 + 测量点
- Error budget policy

### 7. 容量规划

> **参考**：[深入 05](../深入/05-LLM推理服务的容量规划.md)、[Unit 3 · Week 2](Unit3-推理SLO与静默降级/Week2-容量规划.md)、[附录 E · 模板 4](../附录/E-模板库.md)

- Workload 画像（p50 / p95 输入输出 token）
- 三个容量分别计算（Prefill / Decode / KV cache）
- 实例数 + 冗余 + 预算

### 8. 复合系统 Step 预算

> **参考**：[第 4 章](../知识/04-系统架构与复合AI可靠性数学.md)、[Unit 4 · Week 1](Unit4-复合AI可靠性数学/Week1-Step预算与端到端正确率.md)、[附录 E · 模板 5](../附录/E-模板库.md)

- 每个 step 的正确率 p_i（及估值依据）
- 端到端正确率（**必须有数字**）
- Step 数硬上限

### 9. Verifier / Gate 设计

> **参考**：[Unit 4 · Week 2](Unit4-复合AI可靠性数学/Week2-Verifier设计.md)

- ≥3 个 gate 插入点
- 每个 gate 的类型、成本、预期提升
- False positive 风险

### 10. 灰度 / 回滚 / Shadow 流量

> **参考**：[Unit 3 · Week 4](Unit3-推理SLO与静默降级/Week4-灰度回滚决策树.md)

- 灰度决策树（1% → 5% → 25% → 100%）
- Rollback signal（立即 / 需人工 / 观察期）
- Shadow 流量机制

### 11. 数值级故障 Runbook

> **参考**：[第 9 章](../知识/09-工程底座.md)、[科学 03](../科学/03-Quantization为什么有时坏.md)、[Unit 5 · Week 2](Unit5-数值与编译器级调试/Week2-Runbook产出.md)、[附录 E · 模板 6](../附录/E-模板库.md)

- 触发条件（Quick Check）
- 分层诊断流程 + 具体命令
- 常见根因库

### 12. 事故模式映射

> **参考**：[深入 10](../深入/10-AI系统事故模式库.md)

- 把本系统的**已知风险**对照 [深入 10 的 15 个 Pattern](../深入/10-AI系统事故模式库.md)
- 每个 Pattern：是否存在、已做什么预防、还缺什么

### 13. 遗留风险

- 这套方案扛不住什么？
- 什么情况会 degrade 到不可用？
- 组织协作风险（和 ML / 安全 / 业务的接口）

### 14. 决策建议

**给评审会议的总结**（1 页）：

- 这个系统**当前健康度**（A/B/C/D 评级）
- **P0 改进项**（必须做的）
- **P1 改进项**（3 月内）
- **需要业务侧决策的**（不是技术问题）

**A/B/C/D 评级参考**：

| 等级 | 判据（同时满足）| 含义 |
|---|---|---|
| **A** | 全部 SLO 达标 + 红队 0 critical 未关 + ≥3 个正交 verifier + Forbidden 边界完整 | 可继续扩自治 |
| **B** | 1-2 个 SLO 偶有 burn + 红队 ≤2 high 未关 + ≥1 verifier | 维持现状，按 P1 节奏改进 |
| **C** | 多个 SLO 长期 burn 或质量分桶塌陷 + 红队 ≥1 critical 未关 + 0 verifier | 必须按 P0 改进，否则下调自治档 |
| **D** | 任一 forbidden 边界缺失 / 致命三角未砍腿 / 无 eval 覆盖 | **不该上线**，回到设计阶段 |

## Mastery Gate

达到以下标准才算 Capstone mastered：

### 内容完整度
- [ ] 14 节**全部有内容**（不能留空节）
- [ ] **数字 / 指标都有具体值**（不是"很重要"这种抽象词）
- [ ] **图 ≥ 3 张**（至少 Trifecta + Data Flywheel + Step 预算图）

### 深度与可操作性
- [ ] 每节**引用本书对应章节**（不是抽象理论）
- [ ] 产出可**被非作者读懂**（让同事盲读，能获得有效信息）
- [ ] 决策建议部分**能直接用于评审会议**（不是"需要进一步研究"）

### 真实性
- [ ] **基于真实或半真实系统**（不是纸上虚构）
- [ ] 已知风险部分**列出至少 3 条本方案扛不住的情形**
- [ ] **没做 AI reward hacking**（不要把书里的好例子抄一遍交差）

### 流程完整
- [ ] 经过**至少 1 轮 AI 挑错 + 自己修改**（B2 方式）
- [ ] 经过**至少 1 轮红队测试**（见下一节）
- [ ] 让**至少 1 位同事读过**并给反馈
    - **自学者替代**：发到 Hacker News / 微博 / 公司外部 SRE 群盲 review；或用[深入 12](../深入/12-Claude-GPT-Gemini三大模型系列使用指南.md)讲的"两家 judge 投票"——让 Claude + GPT/Gemini 各挑一遍，比单家更接近"同事 review"

## AI 挑错 + 红队提示词

评审包写完后，依次走这两轮。

### 第一轮 · AI 挑错

```
我刚完成了一份 AI 操作生产环境的架构评审包，请按以下维度挑错：

1. **技术漏洞**：数字不一致、计算错误、指标缺失
2. **逻辑漏洞**：某节说 A 但另一节暗示 not A
3. **业务感薄弱**：哪些判断脱离真实业务场景
4. **引用缺失**：应该回链书里章节但没链
5. **过于乐观**：哪些风险估计偏轻
6. **可操作性**：哪些建议"说了等于没说"

请对每条挑错给出**具体段落引用**，不要笼统讲"建议改进"。

不要直接重写。我自己改。
```

### 第二轮 · 红队

```
假装你是一个恶意 / 粗心的工程师，目标是**让这个系统出事故**。
基于我的评审包，设计 5 种攻击路径：

1. Prompt injection 类（不受信输入）
2. Tool 权限滥用类
3. Cost / DoS 类
4. 质量降级类（静默）
5. 组织 / 流程绕过类

对每种：写清楚**攻击步骤**、**我当前的哪些 gate 会挡住**、**哪些不能挡**。

最后给我：**评审包里还应该加什么 gate / 监控 / 流程**。
```

### 第三轮（可选）· 同伴评审

找团队里另一位 SRE / 架构师 / ML 工程师 **盲读**（不先给 context）。10 分钟后问他：

1. 这套方案的**最大风险**是什么？
2. 你会**先砍哪 3 个环节**？
3. 你会**让业务方先决定什么**？

## 团队评审使用方式

Capstone 产出可以直接**作为评审会议材料**，以下是建议流程：

### 会前（异步）
- 发给参会人提前 3 天读
- 附**会前问题列表**（让大家带观点来）

### 会中（60-90 分钟）
- **10 分钟**：作者陈述 § 1 背景 + § 14 决策建议（**不讲技术细节**，节省时间）
- **30 分钟**：逐节 challenge，focus 在 § 2、5、8、11（风险密集区）
- **20 分钟**：达成 action items
- **10 分钟**：记录保留风险 / 下次评审触发点

### 会后
- Action items 进工单系统
- Capstone 文档进团队 wiki
- 季度 revisit 一次
